# 데이터 수집 프로세스 (FRED · Yahoo Finance · KIS)

## 1. 왜 데이터를 수집하나요?

자동매매가 매수/매도 판단을 하려면 **재료(데이터)** 가 있어야 합니다. 이 시스템은 매일 새벽에 다음 세 가지를 모아서 DB에 쌓아둡니다.

| 출처 | 무엇을 받아오나 | 왜? |
|---|---|---|
| **FRED** | 금리, 실업률, GDP, 인플레이션 등 거시 경제 지표 | 시장 전체 분위기 파악 |
| **Yahoo Finance** | S&P500, 나스닥, 환율, 금값, 개별 종목 주가 | 주가 흐름과 외부 자산 상관관계 파악 |
| **KIS (한국투자증권)** | 잔고, 현재가, 일봉, 주문 결과 | 실제 매매와 실시간 가격 |

> 모은 데이터는 모두 `economic_and_stock_data` 테이블에 **날짜별 한 줄(row)** 로 저장됩니다. (60+ 컬럼)

## 4. 단계별 상세 설명

### ① 마지막 수집 날짜 조회 — `economic_service.py:13-34`

```python
response = supabase.table("economic_and_stock_data")
    .select("날짜").order("날짜", desc=True).limit(1).execute()
```

DB에서 가장 최근 날짜를 가져와 **그 다음 날부터** 수집을 시작합니다.

| 결과 | 다음 행동 |
|---|---|
| 마지막 날짜 = 2026-04-16 | 2026-04-17부터 수집 |
| DB가 비어있음 | **2006-01-01부터** 수집 (초기 적재용) |
| 조회 자체가 실패(예외) | **2006-01-01부터** 수집 (안전장치) |

> RLS(Row Level Security)가 켜져 있는데 정책이 없으면 데이터가 있어도 빈 응답이 와서 **2006부터 다시 받는 사고**가 발생합니다. 그래서 `sql/errorfix.sql`로 RLS를 꺼두는 것이 중요합니다.

### ② 미국 장 시간 체크 — `economic_service.py:85-104`

장 중에 데이터를 받으면 **마감되지 않은 가격**이 들어가 버려서 데이터가 일관되지 않게 됩니다. 그래서 한국 시간 **22:30~06:00은 수집을 건너뜁니다.** (`force=True`로 호출하면 무시)

### ③ FRED 데이터 받기 — `stock.py:222-280`

FRED API는 지표마다 **발표 주기**가 다릅니다.

| 빈도(`frequency`) | 해당 지표 예시 |
|---|---|
| `'d'` 일간 | 10년 기대 인플레이션, 장단기 금리차, 국채 수익률 |
| `'w'` 주간 | 금융스트레스지수, 모기지 금리 |
| `'m'` 월간 | 기준금리, CPI, 실업률, 소비자 심리지수 |
| `'q'` 분기 | GDP, 가계 부채 비율 |

각 지표를 받은 뒤, **모두 일간으로 변환**합니다 (`.resample('D').ffill()`). 자세한 건 6번 항목 참고.

### ④ Yahoo Finance 데이터 받기 — `stock.py:283-318`

`yfinance` 라이브러리 대신 **Chart API를 직접 호출**합니다 (`download_yahoo_chart`). 이유는 yfinance가 가끔 차단되거나 응답이 느리기 때문에, 직접 HTTP 요청이 더 안정적이기 때문입니다.

```python
url = f"https://query1.finance.yahoo.com/v8/finance/chart/{symbol}"
```

각 요청 사이에 `time.sleep(1)`을 넣어 **rate limit**(초당 요청 제한)에 걸리지 않게 합니다.

### ⑤ 나스닥 25종목 받기 — `stock.py:301-318`

위와 같은 방식. 한 종목씩 순차로 받고 각 사이 1초씩 대기.

### ⑥ 데이터프레임 병합 — `stock.py:320-330`

```python
all_data_frames = fred_data_frames + yfinance_data_frames + nasdaq_data_frames
result_df = pd.concat(all_data_frames, axis=1, join='outer')
```

- **axis=1**: 옆으로 붙이기 (컬럼 추가)
- **join='outer'**: 어느 한쪽에만 있는 날짜도 모두 살리기

이렇게 하면 "날짜"가 인덱스인 큰 표가 만들어집니다 (한 행 = 하루치 모든 지표).

### ⑦ 결측치 처리 — `stock.py:333-343`

빈 값이나 `'.'`을 채우고, 같은 날짜가 중복되면 마지막 것만 남깁니다. 자세한 건 6번 항목.

### ⑧ DB 저장 — `economic_service.py:147-209`

```python
all_dates = pd.date_range(start=start_date, end=storage_end_date)  # 어제까지
for date in all_dates:
    check = supabase.table("economic_and_stock_data").select("*").eq("날짜", date_str).execute()
    if check.data:
        # 이미 그 날짜 행이 있으면 → null 컬럼만 update
        supabase.table(...).update(update_dict).eq("날짜", date_str).execute()
    else:
        # 없으면 → 새 행 insert
        supabase.table(...).insert(insert_dict).execute()
```

| 저장 규칙 | 이유 |
|---|---|
| **어제까지만 저장** (오늘은 수집은 하되 저장 X) | 장 종료된 확정 데이터만 영구 저장 |
| 같은 날짜 행이 있으면 **null 컬럼만 update** | 기존 값 보존 |
| 없으면 새 행 insert | 점진적으로 누적 |
| 이전 행 데이터로 빈 값 채우기 (`previous_data`) | **수동 forward-fill** — DB 행 단위에서 빈 칸을 직전 날짜 값으로 보충 |

> **시퀀스 어긋남 주의**: `id` 컬럼이 SERIAL/auto-increment인데, DB 백업 복원이나 직접 INSERT로 데이터를 넣으면 시퀀스 카운터가 안 올라가서 `duplicate key` 에러가 납니다. `sql/errorfix.sql`의 시퀀스 재동기화 SQL로 해결합니다.

## 5. KIS API 데이터 수집은 어떻게 다른가?

KIS는 **실시간/주문용**이라 위 흐름과 다르게 동작합니다.

### 5-1. 토큰 인증 — `app/services/auth_service.py` + `balance_service.py:19-127`

| 단계 | 동작 |
|---|---|
| 1) 메모리 캐시 확인 | `_token_cache`에 유효한 토큰이 있으면 그대로 사용 |
| 2) DB 조회 | `access_tokens` 테이블의 가장 최근 토큰 조회 |
| 3) 만료 시 갱신 | KIS `/oauth2/tokenP` 호출하여 새 토큰 발급 |
| 4) 1분 제한 회피 | KIS는 토큰 발급을 **1분에 1회**로 제한 → 락(Lock)으로 막고 대기 |

### 5-2. 주요 호출

| 함수 | 파일 | 용도 |
|---|---|---|
| `get_all_overseas_balances()` | `balance_service.py` | 해외주식 전체 잔고 조회 |
| `get_current_price()` | `balance_service.py` | 종목 현재가 |
| `inquire_psamount()` | `balance_service.py` | 매수가능금액 |
| `order_overseas_stock()` | `balance_service.py` | 매수/매도 주문 (지정가) |
| `get_overseas_daily_price()` | `volume_service.py` | 일봉 (ATR 계산용) |

### 5-3. KIS 데이터의 "수집" 시점

KIS는 **요청할 때마다 그때그때 받아오는** 방식이라 별도 적재는 안 합니다. 단, 일봉 데이터는 ATR(평균 진폭) 계산을 위해 매수 직후 한 번만 받아 `trade_records.atr` 컬럼에 저장합니다 (`scheduler.py:626-637`).

## 6. 결측치 처리: ffill, bfill 쉽게 설명

### 6-1. 왜 결측치가 생기나요?

세 가지 원인이 있습니다.

1. **발표 주기 차이**: FRED의 GDP는 분기마다 한 번만 발표 → 그 사이 90일은 빈 값
2. **주말/공휴일**: 주식 시장이 안 열리는 날에는 가격 데이터가 없음
3. **데이터 종류별 시작일 차이**: 어떤 지표는 2008년부터, 어떤 건 2010년부터 존재

이걸 그대로 두면 ML 모델이나 기술적 분석이 **NaN(빈 값)** 때문에 계산을 못 합니다.

### 6-2. ffill (forward fill) — "앞 값으로 채우기"

> "어제 값이 오늘도 그대로 유지된다고 가정"

```
원본:    [100, NaN, NaN, 105, NaN, 110]
ffill:   [100, 100, 100, 105, 105, 110]
              ↑    ↑         ↑
         앞 값 100을 복사
```

**이 시스템에서 ffill을 쓰는 곳:**

| 파일 | 줄 | 무엇을 채우나 | 왜? |
|---|---|---|---|
| `stock.py` | 270-278 | FRED 월간/주간/분기 지표 → 일간으로 확장 | 분기 지표(GDP)도 매일 사용해야 하니까 마지막 발표값을 그대로 유지 |
| `stock.py` | 337 | 모든 데이터 병합 후 남은 NaN | 주말·공휴일에 주가가 없으면 직전 거래일 가격을 그대로 사용 |
| `stock_recommendation_service.py` | 96 | RSI 계산 후 비거래일 reindex | 주말 RSI는 마지막 거래일 RSI 값 사용 |
| `economic_service.py` | 175-178 | DB 저장 시 빈 컬럼을 이전 행 값으로 채움 | DB 행 단위 forward-fill |

**왜 forward-fill이 안전한가?**
- 주가, 금리, 인플레이션 같은 데이터는 "변동이 없으면 직전 값이 유지된다"고 보는 게 자연스러움
- "미래 값을 가져와서 과거를 채우는 행위"(bfill)는 **미래 정보 누설(look-ahead bias)** 이 되어 백테스트 성능을 부풀림 → 절대 사용 금지

### 6-3. bfill (backward fill) — "뒤 값으로 채우기"

> "내일 값을 가져와서 오늘 빈 칸을 채우기"

```
원본:    [NaN, NaN, 105, NaN, 110]
bfill:   [105, 105, 105, 110, 110]
          ↑    ↑              
        뒤의 105를 복사
```

**이 시스템에서는 거의 안 씁니다.** 이유는 위에서 말한 **look-ahead bias** 때문입니다. 매매 판단을 할 때 "오늘 시점에 미래 값을 알고 있다"고 가정하면 안 되니까요.

| 만약 bfill을 쓴다면 | 이런 상황에서만 |
|---|---|
| 데이터를 단순히 시각화/통계에 쓸 때 | 매매에는 사용 X |
| 가장 최근 데이터가 아직 안 들어왔을 때 (실무에서 거의 없음) | |

### 6-4. ffill만 쓸 때의 부작용과 대처

문제: 주가가 ffill로 채워지면 **"가격 변동이 0인 날"** 이 많이 생깁니다 (주말, 공휴일).

대처: **RSI 계산 시 변동 0인 날을 제외** — `stock_recommendation_service.py:75-79`

```python
trading_series = series[series.diff() != 0].copy()  # 변동이 0인 날 빼기
```

이렇게 하지 않으면 RSI가 비정상적으로 50에 수렴해버립니다 (변동이 없으니까 gain/loss 모두 0).

## 7. 시각적 정리 — 한 행이 만들어지는 과정

```
2026-04-17 데이터를 만든다고 가정:

[FRED]
  T10YIE (10년 기대 인플레이션) → 2.31  (일간 데이터, 그대로 사용)
  GDPC1  (GDP 성장률)            → 27000 (분기 데이터, 1월 발표값을 ffill)
  CPIAUCSL (소비자 물가지수)      → 310   (월간 데이터, 3월 발표값을 ffill)

[Yahoo Finance]
  ^GSPC (S&P 500)               → 5234.5  (4월 16일 종가)
  ^VIX  (변동성 지수)            → 18.3
  GC=F  (금 가격)                → 2384.0

[나스닥 25]
  AAPL → 175.30
  MSFT → 414.20
  NVDA → 902.10
  ...

       ↓ 모두 합치기 (pd.concat axis=1)
       ↓ ffill로 빈 칸 채우기
       ↓

DB에 한 줄로 저장:
┌─────────────┬─────────┬─────────┬───────┬──────┬─────┬──────┬──────┬──────┬─────┐
│   날짜       │ T10YIE  │ GDPC1   │ CPI   │ S&P  │ VIX │  금  │ AAPL │ MSFT │ ... │
├─────────────┼─────────┼─────────┼───────┼──────┼─────┼──────┼──────┼──────┼─────┤
│ 2026-04-17  │  2.31   │  27000  │  310  │ 5234 │18.3 │ 2384 │175.3 │414.2 │ ... │
└─────────────┴─────────┴─────────┴───────┴──────┴─────┴──────┴──────┴──────┴─────┘
```

이 한 줄이 매일 하나씩 추가되어, 나중에 ML 예측 모델과 기술적 분석이 이 데이터를 입력으로 사용합니다.

## 8. 자주 만나는 문제와 해결

| 증상 | 원인 | 해결 |
|---|---|---|
| 2006-01-01부터 다시 수집함 | RLS로 인해 DB 조회가 빈 응답 | `sql/errorfix.sql`로 RLS 비활성화 |
| `duplicate key value violates unique constraint` | id 시퀀스가 max(id)와 어긋남 | `sql/errorfix.sql`의 시퀀스 재동기화 SQL 실행 |
| 미국 장 시간이라 수집 안 함 | 장 중 데이터는 일관성 깨짐 | 정상 동작 (장 마감 후 자동 재수집) |
| Yahoo 일부 종목 실패 | rate limit 또는 일시적 차단 | `time.sleep(1)`로 완화 중. 재실행 시 누락분만 받아옴 |
| FRED 어떤 지표가 갑자기 사라짐 | FRED 시리즈 ID 변경/중단 | `MORTGAGE5US → MORTGAGE30US`처럼 코드에서 대체 (stock.py:36 참고) |

## 9. 한 줄 요약

> **FRED에서 거시 지표 + Yahoo에서 주가/지수 + 나스닥 25종목 → 모두 날짜 기준으로 합쳐서 → 빈 칸은 ffill로 채워서 → 매일 한 행씩 `economic_and_stock_data`에 누적**
