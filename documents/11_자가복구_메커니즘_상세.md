## 1. Self-Healing 이란 무엇인가

### 1-1. 정의

> **"시스템이 운영 중 발생하는 일시적·국지적 실패를 사람의 개입 없이 스스로 감지하고 복구하는 능력"**

### 1-2. 자동매매 시스템에서 왜 절대 필요한가

자동매매는 다음과 같은 **본질적 불확실성**을 안고 24/7 작동합니다:

| 불확실성 | 예시 | 결과 |
|---|---|---|
| 외부 API 일시 장애 | KIS, AlphaVantage, Anthropic 5xx | 매수 못 하거나 데이터 못 받음 |
| 네트워크 지연 | KIS 응답 시 retry 시점에 가격 변동 | 슬리피지 |
| 데이터 누락 | KIS 일봉 API 응답에 거래량 0 | ATR 계산 실패 |
| 시간대 경계 | 서머타임 전환, 자정 경계 | 시간 비교 오류 |
| 부분 체결 | KIS가 100주 주문에 30주만 체결 | DB와 실제 보유량 불일치 |
| 시스템 재시작 | 운영자 코드 패치, 서버 OS 재부팅 | 메모리 캐시 소실 |

→ 사람이 매일 수동 점검해서 고치면 자동매매라 할 수 없음. **시스템이 알아서 복구**해야 진짜 운영 가능.

### 1-3. Self-Healing vs Fail-Fast vs Fail-Close

| 정책 | 동작 | 우리 시스템에서 |
|---|---|---|
| **Fail-Fast** | 에러 즉시 종료 | 사용 안 함 (중단되면 매수 못함) |
| **Fail-Close** | 의심스러우면 거래 차단 | LLM 호출 전체 실패 시 매수 차단 |
| **Self-Healing** | 일시적 실패 자동 복구 | **대부분의 영역에서 사용** |
| Fail-Open | 실패해도 그냥 진행 | 거의 사용 안 함 (위험) |

→ 우리는 **"Self-Healing 우선, 안전 영역에선 Fail-Close"** 조합.

## 2. 우리 시스템의 자가 복구 설계 철학

### 2-1. 5가지 핵심 원칙

```
원칙 1. 검증된 진실 소스(KIS 원장)를 신뢰하고 DB는 보조로 다룬다
원칙 2. 실패 흔적(NULL, 미체결)을 매 사이클 검사하여 복구한다
원칙 3. 알림/로그 실패는 격리하여 본 로직을 막지 않는다
원칙 4. 외부 API 장애는 폴백 모델 + 지수 백오프로 흡수한다
원칙 5. 시간 기반 가드는 실제 데이터 시간을 비교한다 (시계 의존 X)
```

### 2-2. 자가 복구 vs 사람 개입의 경계

| 시나리오 | 자가 복구 | 사람 개입 |
|---|---|---|
| ATR/TP/SL NULL | ✅ 자동 백필 | — |
| KIS 일시적 5xx | ✅ 다음 사이클 재시도 | — |
| 부분 체결 | ✅ holding_quantity 자동 동기화 | — |
| 매수 실패 (가격 형식 오류) | ✅ 다음 시간 재시도 | (코드 패치는 필요) |
| API 키 만료 | ✅ 토큰 자동 갱신 | API 키 재발급은 사람 |
| LLM 전체 실패 | ❌ Fail-Close (매수 차단) | Slack 알림 → 사람 개입 |
| 데이터베이스 다운 | ❌ 시스템 자체가 멈춤 | 사람 |

## 3. 메커니즘 1: ATR 백필

> **이 시스템에서 가장 정교하고 핵심적인 자가 복구 메커니즘**

### 3-1. 무엇을 복구하나

매수 시점에 KIS 일봉 API 호출이 실패하면 `take_profit_price`, `stop_loss_price`, `atr` 컬럼이 **NULL** 로 들어감. 이 상태로 매도 사이클이 돌면 ATR 기반 동적 손절선이 작동 안 하고 폴백(+5%/-7%)으로 떨어짐.

### 3-2. 발견 경위 (실제 사고)

**ADBE (id=14, 2026-04-24 매수)**:
```
log: 어도비(ADBE) 매수 주문 성공
log: 어도비(ADBE) trade_records 저장 완료 (status: buy_ordered)
       ^^^ ATR=... 라인이 없음 = ATR 계산 실패했지만 NULL로 그대로 INSERT
DB:   atr=NULL, take_profit_price=NULL, stop_loss_price=NULL
```

→ 매도 사이클에서 ATR 손절 안 작동, 고정비율 폴백으로 청산 (의도와 다른 보상/위험비 적용).

### 3-3. 복구 메커니즘 동작 원리

**위치**: `app/services/stock_recommendation_service.py:872-906`

**호출 시점**: 매도 스케줄러가 1분마다 `get_stocks_to_sell()` 호출 → 매도 후보 식별 직전에 자동 실행.

**알고리즘**:

```python
# 1. 모든 holding 종목의 trade_records 조회
trade_records_map = supabase.table("trade_records").eq("status", "holding").execute()

# 2. 각 종목별로 ATR 누락 여부 검사
for ticker, tr in trade_records_map.items():
    # 이미 채워져 있으면 스킵 (한 번 복구되면 재호출 X)
    if tr.get("take_profit_price") and tr.get("stop_loss_price"):
        continue

    try:
        # 3. buy_date 기준 일봉 데이터 가져오기 (★ 매수 시점 의도 보존)
        api_excd = EXCHANGE_TO_API.get(TICKER_TO_EXCHANGE.get(ticker, "NASD"), "NAS")
        buy_date_str = tr.get("buy_date") or ""
        buy_ymd = buy_date_str[:10].replace("-", "") if len(buy_date_str) >= 10 else ""
        vol_result = get_overseas_daily_price(api_excd, ticker, gubn="0", bymd=buy_ymd)

        # 4. 일봉 API 실패 시 다음 사이클로 (1분 후 재시도)
        if not (vol_result and vol_result.get("rt_cd") == "0"):
            print(f"  {ticker} ATR 백필 실패: 일봉 API 조회 실패")
            continue

        # 5. ATR 계산
        atr_value = self.calculate_atr(vol_result.get("output2", []))
        if not atr_value:
            print(f"  {ticker} ATR 백필 실패: ATR 계산 None")
            continue

        # 6. buy_price 기준으로 TP/SL 계산 (★ 매수 시점과 동일한 공식)
        buy_price = float(tr.get("buy_price") or 0)
        if buy_price <= 0:
            continue
        tp_price = round(buy_price + atr_value * 2.5, 2)  # 익절 = +ATR×2.5
        sl_price = round(buy_price - atr_value * 1.5, 2)  # 손절 = -ATR×1.5

        # 7. DB 업데이트 + 메모리 dict도 갱신 (★ 같은 사이클의 매도 판단에 즉시 사용)
        supabase.table("trade_records").update({
            "atr": atr_value,
            "take_profit_price": tp_price,
            "stop_loss_price": sl_price,
        }).eq("id", tr["id"]).execute()
        tr["atr"] = atr_value
        tr["take_profit_price"] = tp_price
        tr["stop_loss_price"] = sl_price
        print(f"  {ticker} ATR 백필 완료: ATR={atr_value}, 익절가=${tp_price}, 손절가=${sl_price}")

    except Exception as e:
        print(f"  {ticker} ATR 백필 중 오류: {e}")
```

### 3-4. 5가지 핵심 설계 결정

#### ① **매도 사이클에 통합 (별도 백필 잡 X)**

```
대안 A: 별도 cron job → ATR NULL 행만 정리
대안 B: 매도 사이클이 어차피 1분마다 도니까 거기 통합  ← ★ 채택
```

이유: **별도 잡 없이도 자동 복구**. 매도 모니터링 자체에 ATR 정보가 필요하므로 자연스럽게 같은 시점에 처리.

#### ② **`buy_date` 기준 ATR 재계산**

매수 시점 ATR을 보존하기 위함. 만약 호출 시점(현재) 기준으로 계산하면:
- 매수 후 며칠 지나서 변동성 변하면 의도 어긋남
- 예: 매수 시 ATR=10이었는데 백필 시점엔 ATR=20 → 손절선 너무 넓어짐

→ `bymd=buy_ymd` 인자로 KIS 일봉 API에 매수일자 명시.

#### ③ **메모리 dict 즉시 갱신**

```python
# DB 업데이트 후 메모리 dict도 갱신
tr["atr"] = atr_value
tr["take_profit_price"] = tp_price
tr["stop_loss_price"] = sl_price
```

→ **같은 매도 사이클에서 백필된 값을 즉시 사용 가능**. 다음 1분까지 기다릴 필요 없음.

#### ④ **한 번 복구되면 자동 스킵**

```python
if tr.get("take_profit_price") and tr.get("stop_loss_price"):
    continue
```

→ 백필 완료된 행은 다음 사이클부터 KIS 일봉 API 호출 안 함. **API 누적 부담 0**.

#### ⑤ **API 실패 시 폴백으로 안전 작동**

ATR 백필 실패하면 그 종목은 매도 판단에서 **고정비율 폴백(+5%/-7%)** 사용 (`stock_recommendation_service.py:903-906`):

```python
if trade_record and trade_record.get("take_profit_price") and trade_record.get("stop_loss_price"):
    # ATR 기반 동적 익절/손절
    ...
else:
    # 고정비율 폴백
    if price_change_percent >= 5: 익절
    elif price_change_percent <= -7: 손절
```

→ **백필이 N번 실패해도 시스템은 최소한의 매도 판단 가능**.

### 3-5. 이식 시 체크리스트

다른 시스템에 ATR 백필 패턴을 이식하려면:

- [ ] **데이터 소스의 시점 지정 가능 여부 확인**
  - KIS는 `bymd` 파라미터 지원 → 매수일자 기준 데이터 재조회 가능
  - 다른 API라면 historical data endpoint 있는지 확인
- [ ] **백필 실패해도 동작하는 폴백 로직 확보**
  - 고정 % 손절선 등 절대값 기반 안전망
- [ ] **빈도 결정**: 1분, 5분, 1시간 — 매도 모니터링 주기에 맞춤
- [ ] **메모리 dict 갱신** 잊지 말 것 (같은 사이클 즉시 사용)
- [ ] **로깅** 충분히: 실패 단계마다 메시지 (어디서 막혔는지 추적)

## 4. 메커니즘 2: 주문 정합성 동기화 (Reconcile)

### 4-1. 무엇을 복구하나

자동매매 시스템의 **DB 상태와 증권사 원장의 불일치**를 자동으로 동기화.

### 4-2. 발생 가능한 불일치 시나리오

| 시나리오 | DB | KIS 원장 | 복구 방향 |
|---|---|---|---|
| 매수 체결 완료 | buy_ordered | 보유 중 | DB → holding |
| 매도 체결 완료 | sell_ordered | 보유 0 | DB → sold |
| 부분 체결 | buy_ordered (100주 주문) | 보유 30주 | DB → holding 30주 |
| 매수 미체결 (장 마감) | buy_ordered | 보유 0 | DB → buy_failed |
| 매도 미체결 (장 마감) | sell_ordered | 여전히 보유 | DB → holding 복원 |
| 외부 매수 (KIS HTS 직접) | (없음) | 보유 중 | DB → 자동 INSERT (고아 감지) |
| 시스템 재시작 | (메모리 캐시 초기화) | 보유 중 | DB 재조회 |

### 4-3. 위치 + 호출 빈도

**`app/utils/scheduler.py:155-275`** 의 `_reconcile_orders()`
**호출**: 매도 스케줄러가 1분마다 → `_execute_auto_sell()` 진입 시 호출

### 4-4. 동작 원리 (의사 코드)

```python
def _reconcile_orders(balance_result):
    # 1. KIS 원장에서 현재 보유 종목 + 수량 추출
    kis_holdings = {item.ovrs_pdno: {"qty": int, "item": dict}
                    for item in balance_result.output1}

    # 2. DB에서 활성 레코드 (buy_ordered, sell_ordered, holding) 조회
    active = supabase.table("trade_records").in_("status",
        ["buy_ordered", "sell_ordered", "holding"]).execute()

    tracked_tickers = set()

    # 3. DB 행마다 KIS와 비교
    for record in active.data:
        ticker = record["ticker"]
        kis_qty = kis_holdings.get(ticker, {}).get("qty", 0)
        tracked_tickers.add(ticker)
        status = record["status"]

        if status == "buy_ordered":
            if kis_qty > 0:
                # 매수 체결 → holding 전환
                update(status="holding", holding_quantity=kis_qty)
                # ★ 이 시점에 매수 체결 알림 발송
                notify_buy_filled(...)
            elif is_after_market_close:
                # 장 마감 후에도 미보유 → 미체결로 마킹
                update(status="buy_failed")

        elif status == "holding":
            # 보유수량 동기화 (부분 체결 추가분 반영 등)
            if kis_qty != record.get("holding_quantity"):
                update(holding_quantity=kis_qty)

        elif status == "sell_ordered":
            if kis_qty == 0:
                # 매도 체결 → sold 전환
                update(status="sold")
                # ★ 이 시점에 매도 체결 알림 발송
                notify_sell_filled(...)
            elif is_after_market_close:
                # 장 마감 후 매도 미체결 → holding 복원
                update(status="holding")

    # 4. 고아 감지 (메커니즘 3에서 상세)
    for ticker, info in kis_holdings.items():
        if ticker not in tracked_tickers:
            # KIS는 보유하는데 DB에 없음 → 자동 INSERT
            insert(...)
```

### 4-5. 핵심 설계 포인트

#### ① **KIS 원장이 진실의 단일 출처(Single Source of Truth)**

DB에 buy_ordered 인데 KIS에 보유 0주 → DB가 틀린 것. 항상 **KIS 기준으로 DB를 수정**.

#### ② **장 마감 후 별도 처리**

```python
is_after_market_close = (now_ny.hour > 16) or (now_ny.hour == 16 and now_ny.minute >= 15)
```

장 중 미체결은 **곧 체결될 가능성**이 있어서 그대로 두고, **장 마감 후 16:15 ET 이후**에야 미체결 확정 처리.

#### ③ **부분 체결 자동 처리**

```python
if kis_qty < record.get("quantity"):
    # 100주 주문 → 30주만 체결됨
    update(status="holding", holding_quantity=30)
    # quantity 는 그대로 (주문 의도 보존)
```

#### ④ **재진입 안전성 (Idempotency)**

같은 reconcile 을 여러 번 돌려도 결과 같음. 이미 holding 인 행은 holding 으로 update해도 동일.

### 4-6. 이식 시 고려사항

- 거래소 원장이 단일 진실 소스라는 원칙 확립
- 장 시간/마감 후 분기 처리
- 부분 체결 처리 (체결 수량 ≠ 주문 수량 케이스)
- Idempotent 한 update 로직

## 5. 메커니즘 3: 고아 감지 (Orphan Detection)

### 5-1. 무엇을 복구하나

**KIS 원장에는 보유 중인데 trade_records 에 없는 종목** 자동 등록.

### 5-2. 발생 시나리오

1. **시스템 미운영 중 외부 매수**: 사용자가 KIS HTS에서 직접 매수
2. **DB 저장 실패**: 매수는 KIS 에 들어갔는데 trade_records INSERT 가 네트워크 오류로 실패
3. **DB 복원 후**: Supabase 백업 복원하면서 일부 행 누락

### 5-3. 동작 위치 + 코드

**`app/utils/scheduler.py:_reconcile_orders` 후반부**:


### 5-4. ATR 백필 메커니즘과의 자동 연계

고아 감지로 INSERT 된 행은 **ATR/TP/SL 모두 NULL**. 하지만 다음 매도 사이클에서:
- ATR 백필 메커니즘(메커니즘 1) 이 자동으로 감지
- KIS 일봉 API 호출 → ATR 계산 → TP/SL 채워 넣음

→ **두 메커니즘이 자연스럽게 연계되어 완전 복구**.

### 5-5. 이식 시 주의사항

- `buy_date`: 감지 시점 사용 (실제 매수일은 모름) → 보유 기간 계산이 부정확할 수 있음

## 6. 메커니즘 4: 매수 재시도

### 6-1. 무엇을 복구하나

**같은 날 매수 시도가 일시적 오류로 실패한 경우 같은 날 자동 재시도**.

### 6-2. 발견 경위

**TSLA (2026-05-06 매수 시도)**:
```
21:11 KST: Daily pipeline Step 4 시작 (force=True)
21:12 KST: TSLA 매수 주문 → KIS 거부 (가격 $390.8506 소수점 4자리)
            ↓
```

### 6-4. 두 가지 매수 트리거의 협력

| 트리거 | 시간 | 시간 체크 | 의도 |
|---|---|---|---|

| **B. 정규 자동 매수** | KST 23:30~23:34 | `force=False` | 안전 백업 |

## 7. 메커니즘 5: LLM 모델 폴백 + 재시도

### 7-1. 무엇을 복구하나

**Anthropic API의 일시적 장애** (529 overloaded, 429 rate limit, 503 service unavailable) 자동 복구.

### 7-2. 위치 + 코드

**`app/services/llm_review_service.py`**

```python
MAX_RETRIES = 3
RETRY_DELAYS = [5, 15, 30]  # 5초 → 15초 → 30초 (지수 백오프)
MODELS = ["claude-opus-4-7", "claude-sonnet-4-6"]  # ★ 폴백 모델 체인

# 모델별 재시도: Opus 3회 → Sonnet 3회 (총 최대 6회)
for model in MODELS:
    for attempt in range(MAX_RETRIES):
        try:
            message = client.messages.create(model=model, ...)
            return parse_and_return(message)

        except json.JSONDecodeError:
            # JSON 파싱 실패는 재시도해도 같으니 다음 모델로
            break

        except (anthropic.APIStatusError, anthropic.APIConnectionError) as e:
            error_code = getattr(e, 'status_code', 0)
            delay = RETRY_DELAYS[attempt]

            if error_code in (529, 429, 503):
                # 일시적 장애 → 대기 후 재시도
                time.sleep(delay)
                continue
            else:
                # 4xx 클라이언트 에러는 재시도 무의미
                break

        except Exception as e:
            time.sleep(delay)
            continue

    # 현재 모델 전체 실패 → 다음 모델로 폴백
    if model != MODELS[-1]:
        logger.warning(f"{model} 전체 실패. 폴백 모델로 전환")

# 모든 모델, 모든 재시도 실패 → Fail-Close (매수 차단) + 알림
notify_llm_failure(...)
return {"reviewed_candidates": [], ...}
```

### 7-3. 4단계 방어선

```
1단계: Opus 4.7 시도 1회       ← 정상 케이스 (대부분 성공)
2단계: Opus 4.7 + 5초 대기 후 재시도
3단계: Opus 4.7 + 15초 대기 후 재시도
4단계: Sonnet 4.6 폴백 → 같은 3회 재시도
5단계: 전체 실패 → Fail-Close (매수 차단) + Slack 알림
```

→ 일시적 장애는 1~3단계에서 흡수, 모델 차원의 장애는 4단계에서 흡수, 그래도 안 되면 5단계로 안전 차단.

### 7-4. 핵심 설계 결정

#### ① **에러 종류별 차별 대응**
- 5xx (서버 장애): 재시도
- 4xx 클라이언트 에러: 재시도 무의미 → 즉시 다음 모델로
- JSON 파싱 실패: 재시도해도 같음 → 즉시 다음 모델로

#### ② **지수 백오프**
5초 → 15초 → 30초. **초기 빠른 재시도, 점차 길게 대기**. 일시적 spike 는 5초 안에 해소되는 경우 많음.

#### ③ **모델 다양성 (Diversity)**
같은 모델만 재시도하면 같은 이유로 실패. **다른 모델(Sonnet)로 fallback** 해서 모델 차원 장애도 흡수.

#### ④ **Fail-Close 안전선**
모든 폴백 실패 시 매수 차단 (`reviewed_candidates=[]`). **불확실한 상황에선 거래 안 함이 안전**.

### 7-5. 이식 시 패턴

```python
# 일반화된 외부 API 자가 복구 패턴
def call_external_api_with_self_healing(operation):
    BACKEND_OPTIONS = ["primary", "secondary"]   # 폴백 체인
    MAX_RETRIES = 3
    BACKOFF = [5, 15, 30]

    for backend in BACKEND_OPTIONS:
        for attempt in range(MAX_RETRIES):
            try:
                return operation(backend)
            except TransientError:    # 재시도 가능
                time.sleep(BACKOFF[attempt])
                continue
            except PermanentError:    # 재시도 무의미
                break

    raise FailureAfterAllRetries()
```

## 8. 메커니즘 6: 토큰 자동 갱신 + 캐시

### 8-1. 무엇을 복구하나

**KIS API 토큰 만료 → 자동 갱신**, **1분당 1회 발급 제한 → 락으로 회피**.

### 8-2. 다층 캐시 구조

**`app/services/balance_service.py:19-127`**

```
┌─────────────────────────────────────────┐
│ Layer 1: 메모리 캐시 (_token_cache)      │
│   가장 빠름, 프로세스 재시작 시 소실      │
└─────────────────────────────────────────┘
                  ↓ 캐시 없으면
┌─────────────────────────────────────────┐
│ Layer 2: Supabase access_tokens 테이블  │
│   다소 느림, 프로세스 재시작 후에도 유지   │
└─────────────────────────────────────────┘
                  ↓ DB 토큰도 만료됐으면
┌─────────────────────────────────────────┐
│ Layer 3: KIS API /oauth2/tokenP 호출    │
│   가장 느림, 1분당 1회 제한              │
└─────────────────────────────────────────┘
```

### 8-3. 1분 제한 회피 로직

```python
import threading

_refresh_lock = threading.Lock()
_last_refresh_time = 0

def get_access_token():
    # 1단계: 메모리 캐시 확인
    if cache_valid:
        return cache.access_token

    # 2단계: 1분 제한 체크
    if time.time() - _last_refresh_time < 60:
        wait_seconds = 60 - (time.time() - _last_refresh_time)
        time.sleep(wait_seconds)  # 1분 안 됐으면 대기

    # 3단계: 락으로 동시성 방지 (여러 스레드가 동시 갱신 시도 방지)
    with _refresh_lock:
        # 락 획득 후 다시 캐시 확인 (다른 스레드가 갱신했을 수 있음)
        if cache_valid:
            return cache.access_token

        # DB 조회
        token_data = supabase.table("access_tokens").order("updated_at", desc=True).limit(1).execute()
        if token_valid(token_data):
            cache.access_token = token_data["access_token"]
            return token_data["access_token"]

        # 새 토큰 발급
        new_token = call_kis_token_api()
        cache.access_token = new_token
        save_to_supabase(new_token)
        _last_refresh_time = time.time()
        return new_token
```

### 8-4. 자가 복구 흐름

| 상황 | 대응 |
|---|---|
| 메모리 캐시 만료 | DB 조회 → 사용 |
| DB 토큰도 만료 | 새 발급 |
| 새 발급 시도가 1분 제한에 걸림 | sleep 후 재시도 |
| 동시에 여러 요청이 토큰 갱신 시도 | 락으로 직렬화, 한 번만 발급 |
| 프로세스 재시작 | DB 토큰으로 즉시 복원 |

### 8-5. 이식 시 적용 가능 영역

**모든 OAuth2/Bearer Token 기반 API**:
- 거래소 API (KIS, Interactive Brokers, Alpaca)
- 클라우드 API (AWS, GCP)
- 외부 SaaS (Slack, Notion 등)

→ **3-Layer Cache + Lock + Throttle** 패턴은 범용적으로 활용 가능.

```python
def _send(title, message, color, fields=None):
    if not settings.SLACK_WEBHOOK_URL:
        return False  # ★ Webhook 미설정이면 no-op (예외 X)

    try:
        resp = requests.post(
            settings.SLACK_WEBHOOK_URL,
            json={"attachments": [...]},
            timeout=5,  # ★ 5초 타임아웃 (무한 대기 방지)
        )
        if resp.status_code != 200:
            logger.warning(f"Slack 전송 실패 ({resp.status_code})")
            return False
        return True
    except Exception as e:
        logger.warning(f"Slack 전송 예외: {e}")
        return False  # ★ 예외 삼킴 (호출자에게 전파 안 함)
```

→ **2단 안전망**: 호출자도 try/except, 함수 내부도 try/except. 어느 한 쪽이 깨져도 다른 쪽이 흡수.

### 10-5. 헬퍼 함수도 동일 패턴

`notification_service._get_account_summary()`:

```python
def _get_account_summary() -> str:
    try:
        balance = get_all_overseas_balances()
        if balance.get("rt_cd") != "0":
            return ""  # ★ KIS 실패 시 빈 문자열 (예외 안 던짐)
        ...
        return formatted_string
    except Exception as e:
        logger.warning(f"계좌 요약 조회 실패: {e}")
        return ""  # ★ 어떤 예외든 빈 문자열로 흡수
```

→ 알림 일부 정보가 누락되어도 알림은 발송됨.

### 10-6. 이식 시 격리 패턴

```python
# 본 비즈니스 로직과 부수 작업 분리
def main_business_logic():
    # 1. 핵심 트랜잭션 (절대 실패하면 안 됨)
    result = critical_operation()

    # 2. 부수 작업 격리 (실패해도 1번에 영향 없음)
    try:
        side_effect_1()  # 알림
    except Exception as e:
        log.warning(f"side_effect_1 실패: {e}")

    try:
        side_effect_2()  # 통계
    except Exception as e:
        log.warning(f"side_effect_2 실패: {e}")

    return result
```

→ **핵심 vs 부수 분리** + **부수 작업 try/except 개별 격리**.

## 11. Self-Healing 설계 패턴 — 이식 가이드

위 8가지 메커니즘에서 추출한 **재사용 가능한 6가지 패턴**:

### 패턴 1: 사이클 통합 백필

**언제 사용**: 일시적 데이터 누락이 자주 발생, 별도 cron job 만들기 부담

**구조**:
```python
def main_cycle():
    # 1. 작업 대상 조회
    items = fetch_items()

    # 2. 누락된 항목 백필 (★ 이 부분이 자가 복구)
    for item in items:
        if has_missing_field(item):
            backfilled_value = recompute_or_refetch(item)
            if backfilled_value:
                update_db(item, backfilled_value)
                item.update_in_memory(backfilled_value)  # ★ 즉시 사용

    # 3. 본 작업 진행 (백필된 데이터 활용)
    do_main_work(items)
```

### 패턴 2: 진실 소스 동기화 (Reconcile)

**언제 사용**: 외부 시스템과 내부 DB 가 분리, 정합성 보장 필요

**구조**:
```python
def reconcile():
    external_state = fetch_from_authority()   # ★ 외부 = 진실 소스
    internal_state = fetch_from_db()

    for entity_id in all_entities:
        ext = external_state.get(entity_id)
        intern = internal_state.get(entity_id)

        if ext != intern:
            # 외부 기준으로 내부 갱신
            update_internal(entity_id, ext)

    # 외부엔 있는데 내부엔 없는 것 (Orphan)
    for entity_id in external_state.keys() - internal_state.keys():
        insert_internal(entity_id, external_state[entity_id])
```

### 패턴 3: 폴백 체인 + 지수 백오프

**언제 사용**: 외부 API 호출, 일시적 장애 가능성

**구조**:
```python
BACKENDS = ["primary", "secondary", "tertiary"]
MAX_RETRIES = 3
BACKOFF_SECONDS = [5, 15, 30]

for backend in BACKENDS:
    for attempt in range(MAX_RETRIES):
        try:
            return call(backend)
        except TransientError:
            time.sleep(BACKOFF_SECONDS[attempt])
            continue
        except PermanentError:
            break  # 다음 backend 로
    # 다음 backend 로 폴백
raise FailureAfterAllRetries()
```

### 패턴 4: 다층 캐시 + 동시성 락

**언제 사용**: 비싼 API 호출, 동시 요청 가능

**구조**:
```python
import threading
_lock = threading.Lock()
_memory_cache = {}

def get_resource(key):
    # Layer 1: 메모리
    if key in _memory_cache and not expired(_memory_cache[key]):
        return _memory_cache[key]

    with _lock:
        # 락 획득 후 재확인 (다른 스레드가 갱신했을 수 있음)
        if key in _memory_cache and not expired(_memory_cache[key]):
            return _memory_cache[key]

        # Layer 2: 영구 저장소
        from_db = fetch_from_db(key)
        if from_db and not expired(from_db):
            _memory_cache[key] = from_db
            return from_db

        # Layer 3: 외부 API
        new_value = fetch_from_external_api(key)
        _memory_cache[key] = new_value
        save_to_db(key, new_value)
        return new_value
```

### 패턴 5: 성공 후 상태 갱신

**언제 사용**: 재시도 가능 작업, 한 번 실패해도 같은 윈도우에 재시도 의도

**구조**:
```python
def attempt_action():
    success = False
    try:
        do_actual_work()
        success = True
    except Exception:
        log.error(...)

    if success:
        mark_state_as_done()
    # ↑ 실패 시엔 mark 안 하므로 다음 trigger 에서 재시도
```

### 패턴 6: 부수 작업 격리

**언제 사용**: 본 비즈니스 + 알림/통계/로깅 등 부가 기능

**구조**:
```python
def critical_business_op():
    # 1. 핵심 트랜잭션
    result = main_transaction()

    # 2. 부수 작업 (각각 격리)
    for side_effect in [notification, analytics, logging]:
        try:
            side_effect(result)
        except Exception as e:
            log.warning(f"{side_effect.__name__} 실패: {e}")

    return result
```

## 12. 한계와 미래 작업

### 12-1. 현재 시스템이 자가 복구 못 하는 시나리오

| 시나리오 | 영향 | 사람 개입 필요 이유 |
|---|---|---|
| Supabase 다운 | 시스템 전체 정지 | DB 가 SoT — 복구 불가능 |
| KIS API 키 만료 | 매수/매도 모두 차단 | 키 재발급은 외부 작업 |
| Anthropic API 키 잘못 설정 | LLM 검토 실패 → Fail-Close | 키 교체는 운영자 |
| 코드 자체 결함 | 배포된 후 발견 | 코드 패치 필요 |
| 시장 자체 충격 (서킷브레이커) | 매도 못 함 | 거래소 정상화 대기 |

### 12-2. 추가 자가 복구 메커니즘 후보 (P1)

#### ① **Slippage 자동 회피**
- **현재**: 매도 주문 시 1.7초 시차로 -1.4% 슬리피지 발생 (MSFT 사례)
- **개선**: 손절 시점에 시장가(`ORD_DVSN: "01"`) 자동 전환

#### ② **시간 일관성 자가 검증**
- **현재**: 시스템 시계 기준 NY 시간 계산
- **개선**: NTP 동기화 검증 + KIS 서버 시간 비교

#### ③ **포지션 Sanity Check**
- **현재**: KIS 보유와 DB 만 비교
- **개선**: 매수가 / 현재가 / 평가손익 정합성 자동 검증 (KIS 와 자체 계산값 비교)

#### ④ **데이터 신선도 모니터**
- **현재**: 각 테이블별 갱신 시점 산발적
- **개선**: 모든 테이블 갱신 시각 모니터링 → 24시간 이상 stale 감지 시 알림

#### ⑤ **임계값 자가 튜닝**
- **현재**: V2_BASE_THRESHOLD = 0.4 고정
- **개선**: 최근 7일 BUY 통과 빈도 자동 분석 → 너무 많으면 임계값 ↑, 너무 적으면 ↓

### 12-3. Self-Healing 의 본질적 한계

> **"자가 복구는 일시적·국지적 실패만 다룬다. 영구적·구조적 결함은 사람이 고쳐야 한다."**

| 결함 종류 | 자가 복구 가능? |
|---|---|
| 일시적 API 장애 | ✅ |
| 데이터 누락 | ✅ |
| 시간 경계 오류 | ✅ |
| 코드 로직 결함 | ❌ (사람이 수정) |
| 비즈니스 룰 변경 | ❌ (사람이 수정) |
| 인프라 다운 | ❌ (인프라 복구) |

→ **자가 복구는 "운영 안정성" 확보. 결함 자체를 없애는 게 아님.**
