# 15. 시점별(Point-in-Time) 예측 누적 저장 파이프라인

> 작성일: 2026-05-30
> 목적: ML·감성·종합점수가 **실제로 알파를 더하는지** 검증할 수 있도록, 매일 매수 의사결정의
>       입력(팩터)·출력(점수·게이트)을 그 시점 그대로 누적 저장한다.
> 대상 코드: `app/services/snapshot_service.py`(신규), `app/services/stock_recommendation_service.py`(훅),
>           `backtest/snapshot_analysis.py`(분석), DB `signal_snapshots`(신규 테이블)

---

## 1. 왜 필요한가

기존 백테스트(`backtest/`)는 **기술적 신호만** 재현했다. ML(Transformer)·뉴스 감성·LLM 리뷰는
**과거 시점 예측이 저장돼 있지 않아** 룩어헤드 없이 재현이 불가능했다.

- `stock_recommendations`(기술)·`ticker_sentiment_analysis`(감성)는 매일 **delete→insert 로 덮어씀**
  → 어제의 예측이 사라짐.
- 즉 "그날 ML이 뭐라고 예측했고, 그게 실제로 맞았나"를 사후에 알 수 없었다.

**해결:** 매수 파이프라인이 평가한 모든 종목의 팩터·점수·게이트 결과를 `signal_snapshots` 에
누적 저장 → 시간이 쌓이면 forward return 과 조인해 각 팩터의 예측력(IC)을 측정.

---

## 2. 무엇을 저장하나 (`signal_snapshots`)

매수 파이프라인 1회 실행마다, **평가된 모든 후보**(통과/탈락 모두)를 1행씩 저장.

| 그룹 | 컬럼 |
|---|---|
| 키 | snapshot_date(NY), ticker, stock_name, account_type |
| ML | accuracy, rise_probability, last_price, predicted_price |
| 기술 | golden_cross, rsi, macd, signal, sma20/50, macd_buy_signal, volume_ratio, adx, technical_recommended |
| 감성 | sentiment_score, article_count |
| 점수/게이트 | composite_score, scoring_version, passed_prefilter, passed_threshold |
| 시장 | vix_value |

- **통과·탈락 모두 저장**해야 "게이트가 좋은 종목을 골랐는가"를 비교할 수 있다.
- `passed_prefilter`(RSI≤80 & 기술신호 2개↑), `passed_threshold`(composite ≥ 임계값 = 최종 후보).

### 조인으로 합쳐 보는 시점별 데이터
- **LLM 최종 판정** → `llm_decision_logs` (decision_date, ticker) — 이미 시점별 저장됨
- **실제 매수** → `trade_records` (buy_date, ticker)
- 세 테이블을 (날짜, 티커)로 조인하면 "신호 → 점수 → LLM → 체결 → 실제수익" 전 구간 복원.

---

## 3. 동작 방식

```
매수 파이프라인 (get_combined_recommendations_with_technical_and_sentiment)
   results 조립(ML+기술+감성) → score_and_filter(점수 in-place) → final_results(통과)
        └─> snapshot_signals(results, 통과티커집합, vix)   ← 신규 훅 (return 직전)
                upsert signal_snapshots  (snapshot_date,ticker,account_type 멱등)
```

- **비치명적**: try/except 로 감싸 스냅샷 실패가 매매를 절대 막지 않음.
- **멱등**: 같은 날 재실행해도 UNIQUE 제약 + upsert 로 중복 없음.
- `score_and_filter` 가 후보 dict 를 in-place 채점하므로, prefilter 통과 항목은
  composite_score 를 보유한 채로 스냅샷됨(탈락 항목은 NULL).

`db_backup/schema/create_signal_snapshots.sql` 실행:

```sql
CREATE TABLE IF NOT EXISTS signal_snapshots ( ... );  -- 파일 참조
```

- 테이블이 없으면 스냅샷 upsert 가 조용히 실패(매매엔 영향 없음)하므로 **먼저 생성**할 것.

## 5. 분석 (데이터 누적 후)

```bash
python -m backtest.snapshot_analysis        # forward 20거래일
python -m backtest.snapshot_analysis 10      # forward 10거래일
```

산출:
- **팩터 IC**: rise_probability(ML)·sentiment_score(감성)·composite_score 각각이
  forward 수익률을 예측하는 Spearman 순위상관. `|IC|>0.05` 의미, `>0.1` 강함.
- **게이트 효과**: 최종 후보(passed_threshold) vs 탈락의 평균 forward 수익률 차이.

해석 예:
- `composite_score` IC > 0.05 → 점수체계가 실제로 작동
- `rise_probability` IC 로 **ML의 실제 가치** 판정
- `sentiment_score` IC ≈ 0 → 감성 무용(시스템 v2 가정 "감성 IC 0.01~0.03"과 일치 여부 확인)

⚠️ 통계적 유의성엔 **수백 관측치(날짜×종목)** 필요 → 최소 수개월 누적 후 신뢰.

---

## 6. 한계 / 향후 확장

- 현재 스냅샷은 **ML 게이트(정확도≥80%·상승확률≥3%)를 통과한 후보**가 중심.
  ML 게이트 _자체_의 알파(통과 vs 탈락)를 보려면 **게이트 이전 전체 유니버스**의 ML 예측도
  스냅샷해야 함(향후 확장). 단 후보 내 rise_probability 편차로도 ML 예측력은 측정 가능.
- 룩어헤드 없는 진짜 검증은 **앞으로 쌓이는 데이터**로만 가능(과거 소급 불가).
  그래서 "오늘부터 저장 시작"이 핵심 — 빠를수록 좋다.
