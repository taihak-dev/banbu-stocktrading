### 2-4. 결측치 처리 정책

> **항상 ffill (forward fill) 만 사용. bfill 은 look-ahead bias 위험으로 매매 판단에선 금지.**

- **FRED 월간/주간/분기 지표** → `.resample('D').ffill()` 로 일간 확장 (최신 발표값 유지)
- **Yahoo 주말/공휴일 갭** → 직전 거래일 종가 ffill
- **DB 저장 시 빈 컬럼** → 직전 행(이전 날짜) 값으로 행 단위 ffill (`economic_service.py:175-178`)
- **RSI 계산 시 변동 0인 날 제외** → `series.diff() != 0` 필터 (ffill로 채워진 가짜 거래일이 RSI를 50으로 수렴시키는 문제 방지)

## 3. ML 예측 모델 — Transformer (`predict_colab.py`)

### 3-1. 한 줄 요약

> **"과거 90일치 27종목 가격 + 37개 경제지표 → 4-layer Transformer Encoder × 2 (멀티 모달) → 14일 후 27종목 가격 예측 → 현재가 대비 +X% 변동률(Rise Probability)을 산출"**

### 3-2. 학습/예측 환경

| 항목 | 값 | 비고 |
|---|---|---|
| 학습 위치 | **Kaggle Notebook** (이전엔 Colab 수동) | 무료 GPU(P100/T4) + 스케줄링 가능 |
| 트리거 | FastAPI → Kaggle CLI `kernels push` | `app/services/ml_trigger_service.py` |
| 학습 시간 | 약 5~7분 (P100) | 7분 × 매일 = 월 ~3.5h (무료 한도 30h/주 충분) |
| 결과 저장 | Supabase 직접 INSERT | `predicted_stocks`, `stock_analysis_results` |

### 3-3. 모델 구조 (4-layer Transformer × 2 입력)

```
[ 입력 1: 90일 × 27종목 가격 ]   [ 입력 2: 90일 × 37개 경제지표 ]
            │                                │
            ▼                                ▼
   Transformer Encoder × 4         Transformer Encoder × 4
   (MultiHeadAttention 8개         (Self-Attention + FFN)
    + Residual + LayerNorm + FFN)
            │                                │
       Dense(64, relu)                  Dense(64, relu)
            │                                │
            └────────────► Add ◄────────────┘
                            │
                       Dense(128, relu)
                       Dropout(0.2)
                       GlobalAveragePooling1D    (90 → 1)
                       Dense(27)
                            │
                            ▼
                  [ 14일 후 27종목 가격 ]
```

### 3-4. 하이퍼파라미터

| 파라미터 | 값 |
|---|---|
| `LOOKBACK` | 90일 |
| `FORECAST_HORIZON` | 14일 후 |
| `EPOCHS` | 50 |
| `BATCH_SIZE` | 32 |
| `LEARNING_RATE` | 0.0001 |
| `num_heads` | 8 |
| `ff_dim` | 256 |

### 3-5. 출력 — `Rise Probability (%)` 의 정확한 의미

⚠️ **이름은 "Probability"지만 실제로는 14일 후 예측가의 변동률(%)**:

```
Rise Probability (%) = (예측가 - 현재가) / 현재가 × 100
```

> 음수도 가능 (하락 예측). 매수 후보는 `Rise Probability (%) ≥ 2` 만 통과 (`stock_recommendation_service.py:377`).
>
> ⚠️ LLM 검토 프롬프트에서는 **"예측 상승률"** 으로 라벨링하여 LLM이 "확률"로 오해하지 않도록 수정됨 (이전엔 "상승확률"로 표기되어 LLM이 5%를 "5% 확률"로 잘못 해석하던 버그).

### 3-6. DB 저장

- **`predicted_stocks`** — 일별 예측가/실제가 추적 (시각화용)
- **`stock_analysis_results`** — 종목별 최신 평가 + 추천 (`Stock`, `Accuracy (%)`, `Rise Probability (%)`, `Predicted Future Price`, `Recommendation`, `Analysis`) — 매수 시스템이 실제로 읽는 테이블

> 상세: `documents/05_ML_예측_모델_상세.md`

## 4. 뉴스 감성 분석 — AlphaVantage NEWS_SENTIMENT

### 4-1. 흐름

```
대상 종목 = ML 추천 종목 ∪ 현재 KIS 보유 종목 (set으로 중복 제거)
   ↓ 종목당 1회 호출 (sleep 5초로 rate limit 회피)
AlphaVantage NEWS_SENTIMENT (time_from = 3일 전, limit = 100)
   ↓ relevance_score >= 0.2 인 기사만 채택
   ↓ ticker_sentiment_score 평균 계산
ticker_sentiment_analysis 테이블 INSERT (스냅샷 — 호출 시작 시 전체 삭제)
```

### 4-2. 점수 정규화 (매수 점수용)

```python
sentiment_score = (raw_sentiment + 1) / 2  # [-1, 1] → [0, 1]
```

> ⚠️ 정규화로 인해 **부정 감성도 양수 기여로 변환** → 매수 점수에서는 감점 효과 없음 (단지 작게 가산).
>
> 매도 판단에서는 **원본 점수 사용** → `sentiment < -0.15` 가 매도 신호 트리거 조건 (`stock_recommendation_service.py:959-964`).

### 4-3. 한계

- AlphaVantage 무료 플랜 **일 25회 제한** → 종목 25개 넘으면 일부 누락
- 평균 방식이라 **outlier 1개에 취약** (median 대안 가능)
- 3일 윈도우 고정 (빠른 호재/악재 캡처 늦을 수 있음)

## 5. 매수 알고리즘

### 5-1. 두 가지 점수 시스템 (v1 / v2)

`app/services/scoring_service.py` 에서 환경변수 `USE_SCORING_V2` 로 토글:

```python
USE_SCORING_V2: bool = os.getenv("USE_SCORING_V2", "false").lower() == "true"
```

#### v1 (기존, 단순 가중합)

```
composite_score =
    0.25 × rise_score        ← ML 상승률 점수 (5단계 0.2~1.0)
  + 0.25 × tech_score        ← 기술 지표 (가중합/3.5)
  + 0.20 × sentiment_score   ← 감성 정규화 ([-1,1]→[0,1])
  + 0.15 × volume_score      ← 거래량 비율 (4단계 -0.5~0.6)
  + 0.10 × adx_score         ← ADX (3단계 -0.3~0.4)
  + 0.05 × vix_score         ← VIX (3단계 -0.5~0)

통과선: composite_score >= 0.3
```

> 문제: rise_score 분산이 다른 팩터 합보다 큼 → 사실상 ML 단일 결정. (`documents/10_멀티팩터_변별력_개선_기획.md` 참고)

#### v2 (현재 운영 중, Cross-Sectional Z-Score)

```
매일 후보군 N개에 대해:
  z_factor[i] = (factor[i] - mean) / std        # 후보군 내 표준화
  ±3σ winsorize (극단치 클립)

z_tech[i] = (z_macd[i] + z_sma[i] + z_rsi[i]) / 3.0    # 3개 평균

composite_score =
    0.30 × z_rise        ← ML
  + 0.30 × z_tech        ← 기술 (MACD + SMA + RSI 평균)
  + 0.20 × z_vol         ← 거래량
  + 0.20 × z_adx         ← 추세 강도
  + 0.10 × z_sent        ← 감성
  + (VIX는 시장 메트릭이라 가산 X — 별도 임계값 가산 방식)

통과선 (VIX 적응형):
  V2_BASE_THRESHOLD = 0.4   ← ★ 2026-05-04 0.5 → 0.4 로 하향 조정
```

#### v2 VIX 적응형 임계값 (`scoring_service.py:255-273`)

| VIX 범위 | 임계값 | 의미 |
|---|---|---|
| None or < 20 | **0.40** | 평온장 — 평균 +0.4σ |
| 20~25 | 0.45 | 약간 불안 |
| 25~30 | 0.55 | 불안 |
| 30~35 | 0.70 | 공포 — 매우 선별적 |
| > 35 | 하드블록 | 매수 자체 차단 |

### 5-2. 매수 후보 추출 흐름

```
Step 1. 기술 지표 데이터 조회 (stock_recommendations 테이블)
Step 2. ML 예측 데이터 조회 (Rise Probability >= 2% 사전필터)
Step 3. 감성 데이터 조회 (ticker_sentiment_analysis)
Step 4. 종목별 데이터 통합
Step 5. VIX 조회 (economic_and_stock_data 최신 row)
Step 6. ★ 하드블록 1: VIX > 35 → 모든 매수 차단
Step 7. 종목별 종합점수 계산
   ① ★ 하드블록 2: RSI > 80 → 해당 종목 제외
   ② 사전필터: 기술 신호 (골든크로스 + RSI매수 + MACD) 중 2개 이상 충족
   ③ rise_score, tech_score, ..., vix_score 계산
   ④ composite_score = 가중합 (v1 또는 v2)
   ⑤ ★ 하한선 통과 (v1: 0.3 / v2: 0.4 + VIX 적응)
Step 8. composite_score 내림차순 정렬 후 반환
Step 9. (scheduler) LLM 최종 검토 → BUY/HOLD 판정
Step 10. 종목당 매수가능금액의 10% → 지정가 매수 주문
```

### 5-3. 기술 지표 계산 디테일

| 지표 | 위치 | 핵심 |
|---|---|---|
| **SMA** (20일/50일) | `stock_recommendation_service.py:65-67` | `series.rolling(window=N).mean()` |
| **골든크로스** | — | `SMA20 > SMA50` |
| **RSI** (14일, Wilder's) | `:73-97` | `gain.ewm(alpha=1/14).mean() / loss.ewm(...)` — TradingView 표준 방식 |
| **MACD** (12/26/9) | `:99-105` | `EMA12 - EMA26`, signal = `EMA9(MACD)` |
| **ATR** (14일, Wilder's) | `:107-151` | True Range의 14일 Wilder smoothing |
| **ADX** (14일, Wilder's) | `:153-225` | DI 계산 → DX → 14일 Wilder smoothing |

### 5-4. 거래량 비율 계산 (`:295-312`)

```
volume_ratio = 오늘 거래량 / 직전 5거래일 평균 거래량
```

- 주말/비거래일 필터링: `weekday() < 5` + `tvol > 0`

## 6. LLM 최종 검토 (Claude API)

### 6-1. 역할

> **"매수 후보 추출 후 마지막 게이트. LLM은 거부권만 행사 (BUY → HOLD만 가능). 새 종목 추가 불가, 시스템이 거른 종목 살릴 수 없음."**

### 6-2. 모델 + 폴백 전략

```python
# llm_review_service.py:11-14
MODELS = ["claude-opus-4-7", "claude-sonnet-4-6"]  # Opus 실패 시 Sonnet 폴백
MODELS_WITHOUT_TEMPERATURE = {"claude-opus-4-7"}    # Opus 4.7은 temperature 파라미터 미지원
```

| 우선순위 | 모델 | 동작 |
|---|---|---|
| 1순위 | `claude-opus-4-7` | temperature 미전달 (4.7부터 sampling 파라미터 제거됨) |
| 2순위 (폴백) | `claude-sonnet-4-6` | `temperature=0` 전달 (결정론적) |

각 모델당 **3회 재시도** (5초 → 15초 → 30초 백오프). 처리하는 에러 코드: 429 (rate limit), 503 (service unavailable), 529 (overloaded). 그 외 에러는 즉시 다음 모델로 폴백.

### 6-3. Fail-Close 정책

> **LLM 호출이 전부(Opus 3회 + Sonnet 3회) 실패하면 매수 차단.** Slack 즉시 알림 발송.

`ANTHROPIC_API_KEY` 미설정 시에도 동일하게 매수 차단 + Slack 알림.

### 6-4. 프롬프트 핵심

- **페르소나**: "월스트리트 경력 20년의 미국 주식 트레이딩 전문가이자 최종 의사결정자"
- **검토 기준**:
  1. 기술 지표 검증 (골든크로스 효력 / RSI 과매수 / 추세 약함)
  2. 외부 리스크 (실적 발표 1주 이내 / FOMC·CPI 임박 / 종목 특이 리스크)
  3. 포트폴리오 균형 (같은 섹터 3개 이상 시 가장 약한 종목 HOLD)
- **응답 형식**: JSON 강제 (`market_analysis`, `decisions: [{ticker, decision, reason}]`)

### 6-5. 종목별 입력 데이터 (예측 상승률은 "+X.XX%" 명시)

```
1. 넷플릭스 (NFLX)
   - ML 예측: 예측 상승률 +5.32% (현재가 $93.68 → 예측가 $98.65)
   - 기술적 지표:
     골든크로스: ✓ (SMA20: 92.45, SMA50: 91.20)
     RSI: 56.32 (강세 진입)
     MACD: 0.4521, Signal: 0.3210, 매수신호: ✓
   - 거래량: 5일 평균 대비 1.85배
   - ADX(추세강도): 28.5 (강한 추세)
   - 감성분석: 0.25 (기사 12개)
   - 종합점수: 0.6171
     (예측상승률: 0.6, 기술: 1.0, 거래량: 0.6, ADX: 0.4, VIX: 0)
```

### 6-6. 결정 로그 — `llm_decision_logs` 테이블

`upsert(on_conflict="decision_date,ticker")` — 같은 날 같은 종목 중복 호출해도 마지막 결정만 저장.

| 컬럼 | 의미 |

## 7. 매수 실행 (KIS API)

### 7-1. 매수 시간

| 항목 | 값 |
|---|---|
| 트리거 | NY 시간 **10:30~10:34 ET** (장 시작 후 1시간) |
| KST 환산 (서머타임/EDT) | **23:30~23:34** ← 현재 적용 중 (3월 둘째 일요일 ~ 11월 첫째 일요일) |
| KST 환산 (EST) | 00:30~00:34 (다음날) — 11월 첫째 일요일 ~ 3월 둘째 일요일 |
| 스케줄러 깨어남 | 5분마다 (`schedule.every(5).minutes.do(_run_auto_buy)`) — 시간 윈도우 확인 후 실행 |
| 평일/주말 | 평일만 실행 |
| 실행 위치 | `app/utils/scheduler.py:444-665` (`_execute_auto_buy`) |

### 7-2. 매수 흐름 (스케줄러 트리거 후)

```
1. 매수 시간 + 평일 체크 (NY 시간 기준)
2. KIS 잔고 조회 → 보유/주문 중 종목 set 생성 (중복 매수 방지)
3. 추천 종목 받아오기 (LLM 검토 포함)
4. LLM 검토 통과한 BUY 종목만 진행
5. 종목별 처리:
   a. KIS 현재가 조회 (지정가 매수용)
   b. KIS 매수가능금액 조회 → 10% 분배
   c. 매수 주문 실행 (지정가, 현재가 ± 0)
   d. 성공 시 trade_records INSERT (status=buy_ordered)
      ★ ATR 기반 익절가/손절가 계산:
         take_profit = current_price + ATR × 2.5
         stop_loss   = current_price - ATR × 1.5
      ★ ATR 계산 실패 시 NULL 저장 (매도 단계에서 백필됨)
   e. Slack 매수 체결 알림 + 보유 현황표
   f. 다음 종목까지 sleep(1)
6. KIS 원장과 trade_records 정합성 확인 (_reconcile_orders)
   - 매수 미체결 → buy_failed
   - 매수 체결 → status: buy_ordered → holding
   - KIS 보유 중인데 trade_records 없음 → 고아 감지 → 자동 INSERT (ATR/TP/SL은 NULL)
```

### 7-3. KIS 토큰 관리

| 단계 | 동작 |
|---|---|
| 1. 메모리 캐시 확인 | `_token_cache` 에 유효 토큰 있으면 사용 |
| 2. DB 조회 | `access_tokens` 테이블의 가장 최근 토큰 |
| 3. 만료 시 갱신 | KIS `/oauth2/tokenP` 호출 |
| 4. 1분 제한 회피 | KIS는 토큰 발급 1분당 1회 → Lock으로 막고 대기 |

> ⚠️ `access_tokens` 테이블 스키마: `id`, `token_type`, `access_token`, `expires_at`, `updated_at` (서머타임 자동 처리는 `pytz.timezone('America/New_York')` 사용)

## 8. 매도 알고리즘

### 8-1. 매도 시간

| 항목 | 값 |
|---|---|
| 트리거 | 미국 장 시간 (NY 09:30~16:00 ET) |
| 체크 주기 | **1분마다** |
| KST 환산 (EDT) | 22:30~05:00 |
| 실행 위치 | `app/utils/scheduler.py:272-442` (`_execute_auto_sell`) |

### 8-2. 매도 트리거 (OR 조건)

#### 1. ATR 기반 동적 익절/손절

```
take_profit = buy_price + ATR × 2.5    # 보상/위험비 ≈ 1.67:1
stop_loss   = buy_price - ATR × 1.5
```

- `trade_records.take_profit_price` / `stop_loss_price` 가 채워져 있으면 이 값 사용
- 둘 중 하나라도 NULL 이면 **고정비율 폴백**: +5% 익절 / -7% 손절

#### 2. 기술적 매도 신호 (4개 카운트)

| # | 조건 |
|---|---|
| ① | 데드 크로스 (`SMA20 < SMA50`) |
| ② | RSI > 70 (과매수) |
| ③ | MACD 매도 신호 (`MACD < Signal`) |
| ④ | 패닉셀 (거래량 ≥ 2배 + 당일 ≤ -3% 하락) |

#### 3. 매도 임계치

| 조건 | 매도 발동 |
|---|---|
| 신호 3개 이상 | 매도 |
| 신호 2개 이상 + `sentiment_score < -0.15` | 매도 |
| VIX > 30 + 신호 2개 이상 | 매도 |
| VIX > 40 + 신호 1개 이상 | 매도 |

#### 4. ADX 보정

- ADX > 25 (강한 추세) → 필요 신호 수 1개 차감 (강한 추세에선 약한 신호도 신뢰)

### 8-3. ATR 백필 (Self-Healing) — 2026-04-27 추가

> **매수 시 silent 실패로 ATR/TP/SL이 NULL 저장된 경우, 매도 사이클에서 자동 복구.**

위치: `stock_recommendation_service.py:872-906` (매도 후보 식별 직전)

```python
# 6-1. ATR/익절가/손절가가 누락된 holding 자동 백필
for ticker, tr in trade_records_map.items():
    if tr.get("take_profit_price") and tr.get("stop_loss_price"):
        continue
    try:
        api_excd = EXCHANGE_TO_API.get(TICKER_TO_EXCHANGE.get(ticker, "NASD"), "NAS")
        # ★ buy_date 기준 일봉으로 ATR 계산 (매수 시점 의도 보존)
        buy_date_str = tr.get("buy_date") or ""
        buy_ymd = buy_date_str[:10].replace("-", "") if len(buy_date_str) >= 10 else ""
        vol_result = get_overseas_daily_price(api_excd, ticker, gubn="0", bymd=buy_ymd)
        ...
        atr_value = self.calculate_atr(...)
        buy_price = float(tr.get("buy_price") or 0)
        tp_price = round(buy_price + atr_value * 2.5, 2)
        sl_price = round(buy_price - atr_value * 1.5, 2)
        supabase.table("trade_records").update({
            "atr": atr_value,
            "take_profit_price": tp_price,
            "stop_loss_price": sl_price,
        }).eq("id", tr["id"]).execute()
        # 메모리 dict도 갱신해서 같은 사이클의 매도 판단에 즉시 사용
        tr["atr"] = atr_value
        tr["take_profit_price"] = tp_price
        tr["stop_loss_price"] = sl_price
    except Exception:
        ...  # 폴백으로 진행
```

**효과**:
- 정상 매수 경로 + 고아 감지 경로 모두 자동 복구
- 한 번 백필되면 다음 사이클부터 `continue` 로 스킵 (API 호출 누적 없음)
- API 실패 시 폴백(+5%/-7%)으로 안전하게 동작 (Fail-safe)

### 8-4. 매도 흐름 (`_execute_auto_sell`)

```
1. KIS 잔고 1회 조회 (reconcile + 매도 판단에 재사용)
2. _reconcile_orders() — KIS와 trade_records 정합성 동기화
   - 매수 체결 확인, 매도 체결 확인, 미체결 처리
3. 미국 장 시간 체크 (NY 09:30~16:00)
4. get_stocks_to_sell() 호출 → 매도 후보 추출
   ★ 이 단계에서 ATR 백필 자동 실행
5. sell_ordered 상태 종목 제외 (중복 매도 방지)
6. 종목별 처리:
   a. KIS 현재가 재조회 → 지정가 매도 주문
   b. 성공 시 trade_records UPDATE
      (status=sell_ordered, sell_price, sell_date, sell_reason, profit_loss, profit_loss_pct)
   c. Slack 매도 체결 알림 + 보유 현황표
   d. sleep(2)
```

**보유 현황표** (`format_holdings_table()`): KIS 잔고 기반, 종목/수량/평단가/현재가/손익/수익률 + 합계 (Slack 코드블록 포맷).

`SLACK_WEBHOOK_URL` 미설정 시 모든 함수가 no-op (개발 환경 안전).

> 상세: `documents/09_Slack_연동.md`

| `stock_recommendations` | 기술 지표로 필터링한 추천 종목 | Daily pipeline Step 3 |

### 12-2. `trade_records` 상태 전이

```
buy_ordered  ──KIS 체결 확인──►  holding
                              │
                              ├─ 매도 신호 ─► sell_ordered ─KIS 체결─► sold
                              │
                              └─ 매수 미체결 ─► buy_failed
                              │
                              ▲
                              │
   고아 감지 (KIS 보유 but DB 없음) ─► 자동 INSERT (status=holding, ATR=NULL)
```

### 12-3. `trade_records` 주요 컬럼

| 컬럼 | 의미 |
|---|---|
| `id`, `ticker`, `stock_name`, `exchange_code` | 식별자 |
| `buy_price`, `buy_date`, `quantity`, `holding_quantity` | 매수 정보 |
| `atr`, `take_profit_price`, `stop_loss_price` | ATR 기반 안전망 (NULL 가능 → 백필됨) |
| `composite_score` | 매수 시점 종합 점수 |
| `status` | buy_ordered / holding / sell_ordered / sold / buy_failed |
| `sell_price`, `sell_date`, `sell_reason`, `profit_loss`, `profit_loss_pct` | 매도 시 |
| `account_type` | 모의(`mock`) / 실거래(`real`) |
| `created_at` | 자동 |

### 12-4. RLS 주의사항

> ⚠️ Supabase RLS(Row Level Security)가 켜져 있는데 정책이 없으면 데이터가 있어도 빈 응답이 와서 **2006-01-01부터 다시 수집하는 사고** 발생 가능.
>
> 해결: `sql/errorfix.sql` 의 `ALTER TABLE ... DISABLE ROW LEVEL SECURITY` 실행.

### 12-5. id 시퀀스 어긋남

> ⚠️ `id` 컬럼이 SERIAL/auto-increment인데, DB 백업 복원이나 직접 INSERT로 데이터를 넣으면 시퀀스 카운터가 안 올라가서 `duplicate key` 에러 발생 가능.
>
> 해결: `sql/errorfix.sql` 의 시퀀스 재동기화 SQL 실행.

## 15. 미래 로드맵 — Multi-Factor 변별력 개선

`documents/10_멀티팩터_변별력_개선_기획.md` 의 3-Layer 아키텍처:

```
┌────────────────────────────────────────────┐
│ Layer 1: Cross-Sectional Z-Score (P0)     │  ← ★ 현재 운영 중 (USE_SCORING_V2=true)
│           - 매일 후보군 내 표준화           │
├────────────────────────────────────────────┤
│ Layer 2: Multi-Track 전략 (P1, 1개월)      │  ← 미적용
│           - Momentum (5슬롯, 50%)          │
│           - Mean Reversion (3슬롯, 30%)    │
│           - Quality (2슬롯, 20%)           │
├────────────────────────────────────────────┤
│ Layer 3: Regime-Adaptive 가중치 (P2, 3개월)│  ← 미적용
│           - VIX/Breadth/Trend 분류          │
│           - BULL/NEUTRAL/BEAR/PANIC        │
└────────────────────────────────────────────┘
        + Layer 4 (P3): IC 기반 동적 재가중
```

### 기대 효과 (목표치)

| 지표 | 현재 (추정) | 개선 후 (목표) |
|---|---|---|
| Sharpe Ratio | ~0.8 (강세장 편향) | 1.3+ |
| Max Drawdown | -15~25% | -10% 이내 |
| 팩터 IC 균형도 | ML 0.08, 나머지 0.01~0.03 | 모두 0.03~0.05 |
