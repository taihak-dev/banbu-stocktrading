# 17. 실적 캘린더(Earnings Calendar) 연동 — 구현 기획 / 핸드백 스크립트

> Alpha Vantage `EARNINGS_CALENDAR` API를 일일 데이터 수집 파이프라인에 연동하여,
> 각 종목의 **다음 실적 발표 예정일 + 예상 EPS**를 수집·저장하고,
> **LLM 종목 최종 검토 시 실제 실적일을 참고 정보로 주입**하기 위한 구현 명세서.
>
> 이 문서는 그대로 구현 작업 지시서(핸드백 스크립트)로 사용한다.

> ✅ **구현 완료 (2026-06-27).** 작업 1~8 전부 반영됨. DB 마이그레이션(`sql/create_earnings_calendar.sql`)만 Supabase 콘솔에서 1회 실행 필요.
> 데이터 소스는 **AV 벌크(1차) + Finnhub 폴백(2차, AV 미수록 MU/COST/AVGO 등)** 하이브리드. 키: `ALPHA_VANTAGE_API_KEY_EARNINGS`, `FINNHUB_API_KEY`.
> 관련 파일: `config.py`, `stock_recommendation_service.py`(`preview_earnings_calendar` / `fetch_and_store_earnings_calendar` / `get_combined_...`), `scheduler.py`, `pipeline.py`(`/earnings/preview`·`/earnings/fetch` + Step 1.5), `llm_review_service.py`.

## 0. 배경 — 왜 필요한가

현재 LLM 검토 프롬프트([`app/services/llm_review_service.py:128-129`](../app/services/llm_review_service.py))는
"해당 종목의 실적 발표(Earnings)가 1주일 이내에 예정 → HOLD"라고 지시하지만,
**실제 실적 데이터를 LLM에게 주지 않는다.** LLM은 학습 데이터(컷오프 지남)로 실적일을 *추측*할 뿐이다.

[`documents/06_LLM_검토_로직_상세.md`](06_LLM_검토_로직_상세.md) "14. 알려진 이슈" 표에도
"LLM 학습 컷오프 이후 이벤트 모름"이 본질적 한계로 적혀 있다. 이 구멍을 실제 데이터로 메운다.

## 1. 확정된 설계 결정

| 항목 | 결정 |
|---|---|
| 범위 | `EARNINGS_CALENDAR`만 (발표일 + 예상 EPS). 과거 beat/miss(`function=EARNINGS`)는 범위 밖 |
| LLM 사용 방식 | **참고 정보로만** 제공. "D-X 이내 → HOLD" 같은 강제 룰 문구 금지 |
| API 키 | 전용 키 `ALPHA_VANTAGE_API_KEY_EARNINGS` 사용 (감성분석 키와 분리 → 25/일 예산 충돌 없음) |
| horizon | `6month` (앞으로의 발표일에 버퍼를 두기 위함. 호출 비용 1회로 동일. ⚠️ 단, 우리 유니버스 한정으론 `3month`/`6month`/`12month` 결과가 동일 — 아래 §2 참고) |
| 데이터 소스 | **하이브리드** — 1차: AV `EARNINGS_CALENDAR` 벌크 1회. 2차(폴백): AV 미수록 비-ETF 종목만 **Finnhub** 개별 보강(MU/COST/AVGO 등). (yfinance는 Yahoo 봇차단으로 429 빈발 → Finnhub로 대체. 무료 분당 60회) |
| 실패 정책 | best-effort 비차단 (수집 실패해도 파이프라인/매수 막지 않음) |

### 범위 밖 (이번엔 안 함)
- `function=EARNINGS` 과거 beat/miss 이력 (심볼당 호출 → 무료 한도 초과). 추후 2단계로 분리.

## 2. API 정밀 분석 (실측 기반)

호출 예: `https://www.alphavantage.co/query?function=EARNINGS_CALENDAR&horizon=6month&apikey=...`

| 항목 | 사실 |
|---|---|
| 파라미터 | `function`(필수), `apikey`(필수), `horizon`(`3month`/`6month`/`12month`, 기본 `3month`), `symbol`(선택) |
| 응답 형식 | **CSV** (JSON 아님) |
| 실제 컬럼 (7개) | `symbol, name, reportDate, fiscalDateEnding, estimate, currency, timeOfTheDay` |
| `reportDate` | 발표 예정일 (확정 아닌 추정일 — 종종 변경됨) |
| `estimate` | 애널리스트 EPS 컨센서스. **종종 빈 값** |
| `timeOfTheDay` | pre/post-market. 보통 빈 값 (문서엔 6컬럼이라 적혀있으나 실제론 이 컬럼이 더 옴) |
| 무료 키 한도 | 하루 25회 / 분당 5회 |

### 실측 결과 (2026-06-26, horizon=3month 기준)
- `symbol` 없이 호출 → **전체 미국 시장 6,688행**이 CSV 1회로 수신.
- 우리 유니버스 27개 중 **22개**가 매칭. 누락 5개:
  - **SPY, QQQ** → ETF라 실적 없음 (정상, 영원히 안 잡힘)
  - **COST, MU, AVGO** → Alpha Vantage 무료 피드엔 향후 실적일이 없음(`12month`로도). **→ Finnhub 폴백으로 해결.** 실측(2026-06-27): MU 2026-09-21(EPS 25.27), COST 2026-09-24(6.69), AVGO 2026-09-02(3.30). 핵심 종목 **MU 실적 리스크 정보 확보됨**.

### ⚠️ 함정 정리
1. **CSV 응답.** `csv.DictReader(io.StringIO(response.text))`로 파싱. `response.json()` 금지.
2. **벌크 1회 호출.** `symbol` 없이 전체 시장 받아 우리 종목으로 **로컬 필터**. 종목당 호출(27회) 금지.
3. **컬럼 7개** (`timeOfTheDay` 포함).
4. **ETF(SPY/QQQ)는 실적 없음이 정상** — 에러로 취급하지 말 것.
5. `estimate` 빈 문자열 → `null` 저장.
6. 데모 키/rate-limit 시 CSV 대신 `{"Information":...}` JSON이 올 수 있음 → 헤더에 `symbol` 없으면 경고 로그 후 빈 결과(비차단).

### 작업 2 — Supabase 테이블
`sql/create_earnings_calendar.sql` 신규 (실행은 Supabase 콘솔에서):
```sql
create table if not exists earnings_calendar (
    id bigint generated always as identity primary key,
    ticker text not null,
    company_name text,
    report_date date not null,
    fiscal_date_ending date,
    eps_estimate numeric,          -- estimate 빈 값이면 null
    currency text,
    time_of_day text,              -- timeOfTheDay (pre/post-market, 보통 빈 값)
    fetched_at timestamptz not null default now(),
    unique (ticker, report_date)
);
create index if not exists idx_earnings_ticker_date on earnings_calendar (ticker, report_date);
```

### 작업 3 — 수집 메서드
`app/services/stock_recommendation_service.py`에 [`fetch_and_store_sentiment_for_recommendations:446`](../app/services/stock_recommendation_service.py) 패턴을 미러링.
재사용을 위해 **파싱+필터(저장X)** 와 **저장** 을 두 메서드로 분리:

- 상단에 `import csv, io` 추가 (없으면).
- `preview_earnings_calendar()` — 호출 + 파싱 + 필터, **저장 안 함**, 행 리스트 반환:
  - `requests.get("https://www.alphavantage.co/query", params={"function":"EARNINGS_CALENDAR","horizon":"6month","apikey":settings.ALPHA_VANTAGE_API_KEY_EARNINGS})`
  - `csv.DictReader(io.StringIO(response.text))`. 헤더에 `symbol` 없으면 경고 로그 + `[]` 반환.
  - 유니버스 = `set(STOCK_TO_TICKER.values())` ∪ 현재 보유 티커(`get_all_overseas_balances` output1의 `ovrs_pdno`).
  - 행 매핑: `ticker=symbol`, `company_name=name`, `report_date=reportDate`, `fiscal_date_ending`=빈값→None, `eps_estimate`= `estimate` 빈 문자열이면 None else `float`, `currency`, `time_of_day=timeOfTheDay`.
  - **Finnhub 폴백**: AV 결과에 없는 유니버스 종목(ETF `SPY`/`QQQ` 제외)만 `https://finnhub.io/api/v1/calendar/earnings?from={오늘}&to={+180d}&symbol={t}&token=FINNHUB_API_KEY` 개별 호출 → `earningsCalendar`에서 가장 가까운 미래 `date` + `epsEstimate`, `hour`(bmo/amc/dmh→time_of_day) 매핑해 append. 종목별 try/except, 호출 간 `sleep(1)`. (MU/COST/AVGO 등 AV 미수록 보강)
- `fetch_and_store_earnings_calendar()` — `preview_earnings_calendar()` 호출 → 기존 `earnings_calendar` 전체 delete 후 insert (감성 [:502](../app/services/stock_recommendation_service.py) 방식). `fetched_at`은 DB default.
  - 반환: `{"message": f"{n}개 종목 실적일 저장", "count": n, "results": [...]}`.
  - 전체 try/except로 감싸 예외 시 `{"message": f"실적 수집 실패: {e}", "count": 0}` (비차단).

### 작업 7 — 테스트용 API 엔드포인트 ⭐

**① `GET /pipeline/earnings/preview` — DB 저장 없이 API만 호출 (키/응답 검증용)**

@router.get("/earnings/preview", response_model=dict, summary="실적 캘린더 미리보기 (DB 저장 안 함)")
async def earnings_preview():
    """ALPHA_VANTAGE_API_KEY_EARNINGS 로 EARNINGS_CALENDAR 1회 호출 →

## 5. 수용 기준 (Acceptance)

1. `POST /pipeline/earnings/fetch` 호출 시 AV API **정확히 1회** 발생, `earnings_calendar`에 우리 종목만 저장(실측 22개 + 보유종목).
2. `GET /pipeline/earnings/preview`는 DB 변경 없이 필터된 결과 반환, 잘못된 키면 `count:0` + 안내(예외 아님).
3. `estimate` 빈 값 종목이 `eps_estimate=null`로 저장됨.
4. SPY/QQQ는 "예정 없음"으로 처리되고 에러 없음.
5. LLM 프롬프트에 실제 `D-day` 표기, 데이터 없는 종목은 "예정 없음".
6. 실적 수집을 강제 실패시켜도 **자동/수동 파이프라인 모두** 정상 완료(매수 차단 안 됨).
7. `POST /pipeline/run-buy-pipeline`(수동)으로 실행해도 실적이 **그 시점에 갱신**되어 LLM에 반영됨 (옛 DB 값 안 씀).
8. `llm_decision_logs`에 `earnings_date / days_to_earnings / earnings_estimate`가 함께 기록됨.

## 6. 검증 순서 (구현 후)

1. `GET /pipeline/earnings/preview` → 22개 안팎 정상 반환 확인 (키 동작 검증).
2. `POST /pipeline/earnings/fetch` → Supabase `earnings_calendar` 테이블 확인.
3. `POST /pipeline/run-buy-pipeline` → LLM 프롬프트/로그에 `D-day` 찍히는지 확인.
