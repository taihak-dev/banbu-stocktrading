# 16. 클라우드 배포 및 보안 가이드 (AWS Lightsail · Amazon Linux 2023)

> 로컬에서만 돌던 자동매매 시스템을 **AWS Lightsail (Amazon Linux 2023, 512MB / $5 플랜)** 에 24시간 상시 운영으로 이전하는 전체 절차.
> **실전투자(실제 돈)가 걸린 시스템**이므로, 단순 "올리기"가 아니라 **보안 점검 → 배포 → 상시 실행 → nginx/TLS → 모니터링** 순서로 진행한다.
> OS 기준: **Amazon Linux 2023 (2023.12.20260622.0)** — 사용자명 `ec2-user`, 패키지매니저 `dnf`, 방화벽 `firewalld`.

## 0. 왜 이 구조인가 (사전 이해)

| 항목 | 현황 | 결론 |
|------|------|------|
| 서버 연산량 | FastAPI 앱(`app/`)은 **TensorFlow 미사용**. ML 학습은 Kaggle/Colab에서 별도 실행 | 가장 작은 Lightsail 인스턴스 + swap으로 가능 |
| 상시 실행 필요 | `app/main.py`에서 스케줄러 상시 가동 — **자동매도 1분마다**, 자동매수 5분마다, 일일 파이프라인 KST 21:00 | 잠들면 안 됨 → **24시간 깨어있는 VM 필수** (서버리스/슬립형 무료티어 불가) |
| DB | Supabase(관리형 PostgreSQL) 사용 중 | DB는 추가 작업 불필요. 단 **보안 설정(RLS) 필수** (3장) |
| 비용 | Lightsail 고정 요금제 | **월 $5 (512MB, Dual-stack)**. 첫 3개월 무료 크레딧 |

### 0-1. 요금제 & 메모리 (512MB + swap 전제)
Lightsail은 요금이 **고정**이라 예측 가능하고, 유휴 회수·재고 부족 같은 변수가 없다(Oracle과 차이점).

- 선택: **$5/월 · 512MB RAM · 2 vCPU · 20GB SSD · 1TB 트래픽 (Dual-stack)**
- **실측 메모리**: 라이브러리 import만 한 idle 상태가 약 150MB(pandas+numpy+fastapi+yfinance+supabase). OS까지 합치면 평상시 여유는 200MB 안팎.
- **위험 구간**: 경제데이터 수집 파이프라인(FRED+yfinance+KIS를 pandas로 merge/fill)이 돌 때 메모리가 순간 치솟음 → **swap(2GB)로 OOM 방지**. 이때 잠깐 느려지는 건 감수(매매 자체는 가벼워 타이밍엔 지장 없음).
- 며칠 운영하며 `free -h`로 swap 사용량을 보고, 파이프라인 때 swap을 심하게 먹으면 상위(1GB/2GB) 플랜으로 업그레이드(스냅샷으로 재생성).
- **첫 3개월 무료**: 신규 가입 시 90일 무료 크레딧.

### 0-2. Dual-stack vs IPv6-only — 반드시 Dual-stack
| | Dual-stack (선택) | IPv6-only |
|---|---|---|
| 주소 | 공인 IPv4 + IPv6 | 공인 IPv6만 |
| 의미 | 모든 인터넷과 호환 | IPv4 비용 절약, 단 IPv4 전용 서비스와 통신 제약 |

우리는 **반드시 Dual-stack**:
1. 외부 API(**KIS**·FRED·Yahoo·AlphaVantage)가 IPv4 위주 → IPv6-only면 응답 못 받을 위험.
2. 관리 접속(집 공인 IP `193.186.4.167`)이 IPv4 → IPv6-only 서버엔 집에서 접근 불가할 수 있음.
3. 방화벽 규칙도 전부 IPv4 기준.

## 3. 보안 점검 & 코드 수정 (로컬에서 먼저)

### 3-1. Supabase RLS 활성화 (최우선) 🔴

현재 모든 테이블이 RLS(Row Level Security) 꺼져 있어 **Data API로 외부 노출**된 상태다.

**핵심 개념 — 어떤 키를 쓰고 있는지부터 확인:**
- `anon` 키 (public): RLS 정책의 적용을 받음. 브라우저/클라이언트에 노출돼도 RLS가 막아줌.
- `service_role` 키 (secret): **RLS를 완전히 우회**. 절대 외부 노출 금지. 서버 백엔드 전용.

우리 시스템은 **서버에서만** Supabase에 접근하고(클라이언트 직접 접근 없음) insert/update를 한다. 따라서 **권장 구성**은:

> **`service_role` 키를 서버 `.env`에 두고 + 모든 테이블 RLS는 ON + 외부용 정책은 만들지 않음.**
> 이렇게 하면 서버(service_role)는 정상 동작하고, 외부에서 anon 키로 들어오는 접근은 RLS에 막혀 전부 차단된다.

**적용 방법 (Supabase 대시보드 → SQL Editor):**

```sql
-- 1) 모든 주요 테이블 RLS 활성화
ALTER TABLE economic_and_stock_data       ENABLE ROW LEVEL SECURITY;
ALTER TABLE stock_analysis_results        ENABLE ROW LEVEL SECURITY;
ALTER TABLE stock_recommendations         ENABLE ROW LEVEL SECURITY;
ALTER TABLE ticker_sentiment_analysis     ENABLE ROW LEVEL SECURITY;
ALTER TABLE access_tokens                 ENABLE ROW LEVEL SECURITY;
ALTER TABLE stocks                        ENABLE ROW LEVEL SECURITY;
ALTER TABLE trade_records                 ENABLE ROW LEVEL SECURITY;
ALTER TABLE signal_snapshots              ENABLE ROW LEVEL SECURITY;
-- (그 외 프로젝트에 존재하는 모든 테이블에 대해 동일하게 실행)
```

> RLS를 켜되 **정책(policy)을 하나도 만들지 않으면**, anon 키로는 읽기·쓰기 모두 0건이 된다(완전 차단). `service_role` 키는 RLS를 우회하므로 서버는 그대로 동작한다.

**확인:**
```sql
-- RLS가 꺼진 public 테이블이 남아있는지 점검 (결과가 없어야 정상)
SELECT tablename
FROM pg_tables
WHERE schemaname = 'public' AND rowsecurity = false;
```

**서버 `.env` 키 교체:**
- 현재 `SUPABASE_KEY`가 anon 키라면 → `service_role` 키로 교체 (Dashboard → Project Settings → API → `service_role` `secret`).
- ⚠️ service_role 키는 절대 git/프론트/로그에 노출 금지. 서버 `.env`(권한 600)에만 둔다.

> **추가 권장**: Supabase Dashboard → Settings → API → "Data API" 설정에서, 외부에서 쓸 일이 없다면 노출 스키마를 최소화하거나 Data API 자체 제한도 검토.

### 3-2. API 인증 추가 🔴

`app/api/routes/`에 `balance`, `stocks`, `pipeline` 등 엔드포인트가 있고 **인증이 없다.** Lightsail은 공인 IP가 항상 노출되므로, 8장 nginx(IP 제한+TLS+BasicAuth)로 1차 차단하더라도 **앱 레벨 API Key 인증을 추가로 두는 것**(다층 방어)을 권장한다.

```python
# app/core/auth.py (신규)
from fastapi import Header, HTTPException
from app.core.config import settings

async def require_api_key(x_api_key: str = Header(default="")):
    if not settings.API_KEY or x_api_key != settings.API_KEY:
        raise HTTPException(status_code=401, detail="Unauthorized")
```
```python
# config.py에 추가
API_KEY: str = os.getenv("API_KEY", "")
```
```python
# 보호할 라우터에 의존성 추가 (예: balance, pipeline)
from app.core.auth import require_api_key
router = APIRouter(dependencies=[Depends(require_api_key)])
```

> 모니터링용 `/health` 같은 무인증 엔드포인트는 주문 라우터와 분리해서 별도로 둔다(9-2 참고).

### 3-3. CORS 제한 🟠

`app/main.py`의 `allow_origins=["*"]` + `allow_credentials=True` 조합은 위험하고 사실 브라우저에서 막히기도 한다. 외부 프론트가 없다면 CORS를 닫거나 특정 도메인만 허용:

```python
# app/main.py
app.add_middleware(
    CORSMiddleware,
    allow_origins=[],                      # 외부 프론트 없으면 비움
    # allow_origins=["https://내도메인"],  # 있으면 해당 도메인만
    allow_credentials=False,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

### 3-4. 로컬 git remote의 노출 토큰 🟠

현재 로컬 origin URL에 GitLab PAT(`glpat-...`)가 평문으로 박혀 있다. 이번 배포는 **rsync로 코드만 옮기고 `.git`·`.git-credentials-info`를 제외**(6-1)하므로 토큰이 서버로 가지는 않지만, **로컬에 노출된 PAT 자체가 위험**하다.

- 가능하면 기존 노출 PAT는 **폐기(revoke) 후 재발급**.
- 나중에 서버에서 직접 `git`을 쓸 계획이면, 그때 **읽기 전용 Deploy Token**(`read_repository`)을 별도 발급해 사용.

### 3-5. 시크릿 파일(.env) 관리 🟠

- `.env`는 이미 `.gitignore`에 포함됨(양호). **절대 커밋 금지.**
- 서버 전송 시 git이 아닌 `scp`로 직접 전송하고 권한을 잠근다(6-2).
- 서버의 `.env` 권한: `chmod 600 .env` (소유자만 읽기).
- 운영 전환 시 `.env`의 `KIS_USE_MOCK`을 의도대로 설정했는지 재확인 (모의=`true`, 실전=`false`).

## 4. AWS Lightsail 인스턴스 생성 (Amazon Linux 2023)

### 4-1. 인스턴스 만들기
1. AWS 콘솔 → **Lightsail** → **Create instance**
2. **Region/AZ**: `Seoul (ap-northeast-2)` 권장 (한국에서 가깝고 KIS 지연 낮음)
3. **Platform**: **Linux/Unix**
4. **Blueprint**: **OS Only → Amazon Linux 2023** (앱만 직접 올리므로 "OS Only". WordPress 같은 앱 블루프린트 아님)
5. **Networking type**: **Dual-stack** 선택 (0-2 참고. IPv6-only 금지)
6. **SSH key pair**:
   - 기본 제공 키를 다운로드하거나, **내 키 업로드** 권장. 로컬에서 키 생성(없다면):
     ```bash
     ssh-keygen -t ed25519 -f ~/.ssh/lightsail_trading -C "trading-server"
     ```
     → "Upload new" 로 `~/.ssh/lightsail_trading.pub` 등록.
7. **Instance plan**: **$5/월 · 512MB** (0-1 참고)
8. **Identify your instance**: 이름 예) `stocktrading-prod`
9. **Create instance** → 1~2분 후 **Running**

> Amazon Linux 2023 기본 사용자명은 **`ec2-user`** (Ubuntu의 `ubuntu`와 다름).

### 4-2. 고정 IP(Static IP) 연결 — 필수
Lightsail 기본 공인 IP는 인스턴스를 중지/재시작하면 바뀐다. **고정 IP를 연결**해야 방화벽 규칙·nginx·DNS가 안 깨진다.

1. Lightsail → **Networking** → **Create static IP**
2. Region 동일(Seoul) → 위에서 만든 인스턴스에 **Attach**
3. 부여된 **고정 공인 IPv4**를 메모 (이하 `<서버IP>`)

> 고정 IP는 **실행 중인 인스턴스에 붙어 있는 동안 무료**. 인스턴스에 연결 안 한 채 방치하면 소액 과금되니 주의.

## 5. 서버 초기 설정

### 5-1. SSH 접속
```bash
chmod 600 ~/.ssh/lightsail_trading           # 키 권한 (최초 1회)
ssh -i ~/.ssh/lightsail_trading ec2-user@<서버IP>
# Amazon Linux 2023 기본 사용자명: ec2-user
```

### 5-2. 방화벽 — **내 IP만 허용**

> 🔒 **현재 내 공인 IP: `193.186.4.167`** — SSH(22)를 이 IP에서만 허용한다(SSH 터널로 접근하므로 22만 열면 됨).
> Lightsail은 공인 IP가 상시 노출되므로 출발지 IP 제한이 가장 강력한 1차 방어다.

**(a) Lightsail 방화벽 (클라우드 레벨) — 주 방어선, 필수**
인스턴스 → **Networking 탭 → IPv4 Firewall**. 우리는 **SSH 터널(8-0)** 로 접근하므로 **22번만** 내 IP로 열면 된다:

| Application | Protocol | Port | Restrict to IP (Source) |
|-------------|----------|------|--------------------------|
| SSH | TCP | 22 | `193.186.4.167` |

- 각 규칙에서 **"Restrict to IP address"** 체크 → 내 IP 입력.
- **80·443·8000은 열지 않는다.** 8000은 어차피 `127.0.0.1` 바인딩(7-1)이라 터널로만 들어오고, 80/443은 nginx+TLS(8-1~8-6)를 쓸 때만 필요하다.
- (도메인을 마련해 TLS 경로로 갈 경우에만) 443을 내 IP로, 80을 발급/갱신 때 임시 개방.
- 💡 **복구 경로**: Lightsail 콘솔의 **브라우저 기반 SSH("Connect")** 는 이 클라우드 방화벽을 우회한다. 그래서 내 IP가 바뀌어 22번이 막혀도, 콘솔에서 브라우저 SSH로 들어가거나 방화벽 규칙의 IP만 고치면 바로 복구된다.

**(b) 서버 호스트 방화벽 (firewalld) — 선택적 다층 방어**
AL2023은 `firewalld`를 쓴다(Ubuntu의 ufw 아님). **주의: 호스트 방화벽은 브라우저 SSH도 차단**하므로, 잠그려면 신중히. Lightsail 방화벽만으로도 충분하며, 추가 강화를 원할 때만 적용:

```bash
sudo dnf install -y firewalld
sudo systemctl enable --now firewalld

# 기본 public 존에서 ssh 서비스 제거 (아래 rich rule로 내 IP만 22 허용)
# (dhcpv6-client는 dual-stack IPv6 주소 취득에 쓰일 수 있어 건드리지 않는다)
sudo firewall-cmd --permanent --zone=public --remove-service=ssh

# 내 IP에서만 22 허용 (SSH 터널만 쓰므로 22만. TLS 경로면 443 행도 추가)
sudo firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" source address="193.186.4.167/32" port port="22" protocol="tcp" accept'

sudo firewall-cmd --reload
sudo firewall-cmd --list-all
```
> ⚠️ 호스트 방화벽까지 내 IP로 잠그면, IP가 바뀌었을 때 **브라우저 SSH로도 못 들어온다.** 이 경우 복구는 콘솔에서 인스턴스를 **Stop→Start**(고정 IP라 IP는 유지) 후 잠깐 firewalld가 새로 적용되기 전 접속하거나, 스냅샷 복구가 필요해 번거롭다. **자신 없으면 (b)는 생략하고 Lightsail 방화벽(a)만 사용**을 권장.

> ℹ️ **외부 API 응답은 whitelist 불필요.** KIS/FRED/Yahoo/AlphaVantage/Supabase/Anthropic/Slack 호출은 전부 **아웃바운드(나가는) 연결**이고, 그 응답은 stateful 방화벽이 자동 통과시킨다. 인바운드만 잠그면 된다.

> ⚠️ **가정용 인터넷 IP는 바뀔 수 있다(동적 IP).** 막히면: `curl https://api.ipify.org`로 새 IP 확인 → **Lightsail 콘솔**에서 방화벽 IP 수정(브라우저 SSH로 복구 가능) → firewalld도 쓴다면 rich rule 갱신.

### 5-3. swap 추가 (메모리 보호) 🔴
512MB RAM은 pandas/numpy 파이프라인에서 부족할 수 있다. **swap 2GB**를 추가해 OOM(메모리 부족 크래시) 방지.
> AL2023의 루트 파일시스템은 **xfs**라 `fallocate`로 만든 swap은 `swapon`이 거부할 수 있다(`swapfile has holes`). AWS 공식 방식대로 **`dd`** 로 만든다.
```bash
sudo dd if=/dev/zero of=/swapfile bs=128M count=16   # 128M × 16 = 2GB
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
free -h   # Swap 2.0Gi 확인
```
> swap은 RAM이 아니라 20GB SSD에서 떼어 쓰는 것이라 512MB 인스턴스에서도 문제없이 동작한다. OS+코드+venv(~5GB)를 빼도 디스크는 충분히 남는다.
> (선택) 메모리가 빠듯하므로 swap을 조금 더 적극적으로 쓰게 하려면: `echo 'vm.swappiness=30' | sudo tee /etc/sysctl.d/99-swap.conf && sudo sysctl -p /etc/sysctl.d/99-swap.conf`

### 5-4. 시간대 설정 (스케줄러가 KST 기준이므로 권장)
```bash
sudo timedatectl set-timezone Asia/Seoul
timedatectl   # Asia/Seoul 확인
```

## 6. 코드 배포

### 6-1. 로컬 코드 서버로 전송 (rsync) 🔴
> `git clone`이 **아니라** 로컬 디렉터리를 그대로 복사한다. 그래야 아직 커밋/push 안 한 파일(`app/services/snapshot_service.py`, `backtest/`, `db_backup/schema/*.sql`, 미push 커밋 등)까지 함께 올라간다.
> 단순 `scp -r`은 `venv`·`.venv`(합쳐 ~2GB)·`.git`·42MB 로그까지 다 복사해 느리고 깨진다. **불필요/플랫폼 의존/민감 파일을 제외할 수 있는 `rsync`** 를 쓴다(맥에 기본 탑재).

**(1) 서버에 대상 디렉터리 생성:**
```bash
ssh -i ~/.ssh/lightsail_trading ec2-user@<서버IP> 'mkdir -p ~/stockTrading'
```
**(2) 로컬 PC(맥)에서 rsync 실행:**
```bash
rsync -avz --progress \
  -e "ssh -i ~/.ssh/lightsail_trading" \
  --exclude='venv/' --exclude='.venv/' --exclude='.venv_predict/' \
  --exclude='.git/' --exclude='.git-credentials-info' \
  --exclude='__pycache__/' --exclude='*.pyc' --exclude='*.pyo' \
  --exclude='*.log' --exclude='.DS_Store' \
  --exclude='.env' \
  --exclude='lecture/' \
  /Users/jhim000/Desktop/stockTrading/ \
  ec2-user@<서버IP>:~/stockTrading/
```
- ⚠️ **source 끝의 슬래시(`/`) 필수** — `stockTrading/`는 "디렉터리 *내용*을" 대상에 넣는다는 뜻(슬래시 없으면 `stockTrading/stockTrading/`로 중첩됨).
- 제외 이유: `venv/.venv`는 맥용이라 서버에서 못 씀(6-3에서 서버용 새로 생성) · `.git`/`.git-credentials-info`는 불필요+토큰 노출 · `*.log`는 42MB 불필요 · `.env`는 보안상 따로(6-2) 전송.
- 코드 수정 후 다시 올릴 때도 **같은 명령** → 바뀐 파일만 빠르게 동기화(7-4 재배포).

> 참고: `rsync`가 없거나 1회성이면 `scp -r`도 되지만, 위 제외가 안 돼 ~2GB가 통째로 전송되니 비권장.

### 6-2. `.env` 안전 전송 (rsync에서 제외했으므로 별도로)
```bash
# 로컬 PC에서 실행 (서버로 직접 복사)
scp -i ~/.ssh/lightsail_trading \
    /Users/jhim000/Desktop/stockTrading/.env \
    ec2-user@<서버IP>:~/stockTrading/.env
```
서버에서 권한 잠금 + 내용 점검:
```bash
chmod 600 ~/stockTrading/.env
# KIS_USE_MOCK, SUPABASE_KEY(service_role인지), API_KEY 등 운영값 확인
```

### 6-3. 가상환경 + 의존성
```bash
cd ~/stockTrading
python3.11 -m venv venv
source venv/bin/activate
pip install --upgrade pip
# 512MB라 메모리/디스크 절약 위해 캐시 비활성 권장
pip install --no-cache-dir -r requirements.txt
# TensorFlow는 서버에서 불필요(주석 처리돼 있음) → 설치 안 함
# 만약 설치 중 'Killed'(OOM)가 뜨면 swap(5-3)이 적용됐는지 먼저 확인
```

```bash
source venv/bin/activate

# 로그에 스케줄러 시작 메시지 확인 후 Ctrl+C로 종료
```

## 7. 상시 실행 (uvicorn 로컬 바인딩 + systemd)

**uvicorn은 외부(0.0.0.0)가 아니라 127.0.0.1에만 바인딩**한다. 그러면 8000 포트는 서버 내부에서만 열려, 외부에선 오직 **SSH 터널(8-0)** 을 통해서만 접근된다. (나중에 도메인+nginx로 가면 nginx가 443에서 받아 `127.0.0.1:8000`으로 프록시)

### 7-1. systemd 서비스 파일 생성
```bash
sudo nano /etc/systemd/system/stocktrading.service
```
```ini
[Unit]
Description=Stock Trading FastAPI + Scheduler
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=ec2-user
WorkingDirectory=/home/ec2-user/stockTrading
# 127.0.0.1 바인딩 → 외부 직접 접근 차단, nginx 경유만 허용
ExecStart=/home/ec2-user/stockTrading/venv/bin/uvicorn app.main:app --host 127.0.0.1 --port 8000
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
```
> ⚠️ **`EnvironmentFile`을 일부러 넣지 않았다.** 이 앱은 `app/core/config.py`가 `load_dotenv()`로 `.env`를 직접 읽는다(WorkingDirectory가 프로젝트 루트라 `./.env`를 찾음). 게다가 우리 `.env`에는 `SLACK_NOTIFY_LEVEL=info  # ...`처럼 **인라인 주석**이 있는데, systemd `EnvironmentFile`은 인라인 주석을 못 걸러서 값에 `# ...`까지 들어가 버린다. 그래서 systemd로 주입하지 않고 앱이 직접 로드하게 둔다.
> `run.py`는 `0.0.0.0`으로 바인딩하므로 운영에선 위처럼 **uvicorn을 직접 호출**해 `--host 127.0.0.1`을 강제한다. (`run.py`를 수정해 host를 127.0.0.1로 바꿔도 됨)
> `MemoryMax`는 일부러 설정하지 않았다 — 512MB에선 한도를 잘못 잡으면 파이프라인 도중 앱이 강제 종료될 수 있다. OOM 방지는 **swap(5-3)** 이 맡고, 혹시 죽어도 `Restart=always`가 되살린다.

### 7-2. 등록 & 시작
```bash
sudo systemctl daemon-reload
sudo systemctl enable stocktrading      # 부팅 시 자동 시작
sudo systemctl start stocktrading
sudo systemctl status stocktrading      # active (running) 확인
```
> ⚠️ **기동 직후 바로 응답하지 않을 수 있다.** 이 앱은 시작 시 `startup()`에서 **경제데이터 수집을 먼저 끝낸 뒤** 요청을 받기 시작한다(`app/main.py`의 lifespan). 그래서 `systemctl start` 직후 몇 분간 `curl`이 안 될 수 있으니, 먼저 로그로 진행 상황을 본다:
```bash
journalctl -u stocktrading -f                 # "경제 데이터 수집 완료" 로그가 뜬 뒤
curl -s http://127.0.0.1:8000/                # 서버 내부에서 응답 확인 (JSON 환영 메시지)
```

### 7-3. 로그 보기
```bash
journalctl -u stocktrading -f           # 실시간 로그
tail -f ~/stockTrading/stock_scheduler.log   # 앱 자체 로그
```

### 7-4. 코드 업데이트 루틴 (재배포)
로컬에서 코드 수정 후 → **6-1의 rsync를 다시 실행**(바뀐 파일만 빠르게 전송) → 서버에서:
```bash
cd ~/stockTrading
source venv/bin/activate && pip install --no-cache-dir -r requirements.txt   # requirements 바뀐 경우에만
sudo systemctl restart stocktrading
```
> rsync는 기본적으로 서버에만 있는 파일을 지우지 않는다. 로컬에서 **삭제한 파일까지 서버에 반영**하려면 rsync에 `--delete`를 추가(단, 서버에서 생성되는 `stock_scheduler.log` 등이 지워지지 않도록 제외 목록 유지에 주의).

## 8. 외부 접근 방법 (택1)

> ✅ **우리는 도메인이 없으므로 → 8-0의 SSH 터널을 사용한다. 8-1~8-6(nginx+TLS)는 건너뛴다.**
> ❗ **Let's Encrypt는 "맨 IP 주소"로는 인증서를 발급하지 않는다**(고정 IP라도 도메인 필수). 그래서 도메인 없이 정식 TLS는 불가능하고, SSH 터널이 가장 안전·간단하다.

### 8-0. (권장 · 도메인 불필요) SSH 터널 -> 우린 도메인이 없기 때문에 해당 방식 채택
우리 시스템은 자동매매를 **스케줄러**가 수행하므로 API를 외부에 공개할 필요가 없다. 관리/문서 확인이 필요할 때만 **로컬 PC에서 SSH 터널**로 접속하면 된다 — 인증서·도메인·nginx 전부 불필요.

```bash
# 로컬 PC에서 (서버의 8000을 내 PC 8000으로 안전하게 끌어옴)
ssh -i ~/.ssh/lightsail_trading -L 8000:127.0.0.1:8000 ec2-user@<고정IP>
# 접속 유지한 채 브라우저에서: http://localhost:8000/docs
```
- SSH로 암호화되므로 HTTPS와 동등하게 안전하다.
- 이 방식이면 **Lightsail 방화벽은 22(SSH)만 열면 되고 80/443은 모두 닫는다** → 공격면 최소. (5-2 표에서 HTTPS/443 행 삭제)
- uvicorn은 이미 `127.0.0.1`에만 바인딩(7-1)돼 외부에서 8000 직접 접근 불가. 터널을 통해서만 들어온다.
- 고정 IP라 위 명령의 `<고정IP>`가 안 바뀌어 매번 그대로 쓰면 된다.

> 💡 자주 쓰면 로컬 `~/.ssh/config`에 별칭을 만들어 `ssh trading` + `-L`로 단축 가능.

---

### (참고) 도메인이 생기면: nginx + TLS — 아래 8-1~8-6
> 지금은 **불필요**. 나중에 도메인(또는 DuckDNS 같은 무료 서브도메인)을 마련하면 그때 아래를 따른다.
> ⚠️ 순서 주의: 인증서 없이 `ssl_certificate` 경로를 먼저 적으면 `nginx -t`가 실패한다. **(1) HTTP 프록시 → (2) certbot이 TLS 자동 주입 → (3) 접근제한** 순.

### 8-1. 도메인 준비 + 80 임시 개방
- TLS(Let's Encrypt) 발급에는 도메인이 필요하다. 보유 도메인의 **A 레코드를 `<서버IP>`로** 지정 (예: `trading.mydomain.com`). 전파(수 분~수십 분) 확인: `dig +short trading.mydomain.com` (`dig`가 없으면 `sudo dnf install -y bind-utils`, 또는 `getent hosts trading.mydomain.com`).
- certbot의 도메인 검증(HTTP-01)은 80포트로 들어온다. **발급 동안 Lightsail 방화벽에서 80(HTTP)을 소스 IP 제한 없이 임시 개방**한다(인스턴스 → Networking → IPv4 Firewall → HTTP/80 추가).
- 도메인이 없거나 API 외부 공개가 불필요하면 → 8-6의 **SSH 터널** 대안으로 가고 8-1~8-5는 건너뛴다.

### 8-2. HTTP 리버스 프록시 먼저 구성 (SSL 없이)
> AL2023의 nginx는 Ubuntu식 `sites-available/sites-enabled`가 아니라 **`/etc/nginx/conf.d/*.conf`** 를 자동 include 한다.
```bash
sudo nano /etc/nginx/conf.d/stocktrading.conf
```
```nginx
server {
    listen 80;
    server_name trading.mydomain.com;     # ← 본인 도메인으로 교체

    location / {
        proxy_pass         http://127.0.0.1:8000;
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
    }
}
```
검사 & 기동:
```bash
sudo nginx -t                  # 문법 OK 확인
sudo systemctl enable --now nginx
```
> ⚠️ AL2023 기본 `/etc/nginx/nginx.conf` 안에 `listen 80 default_server` 블록이 있다. `nginx -t`에서 conflict 경고가 나면 nginx.conf의 그 기본 `server { ... }` 블록을 주석 처리하고 다시 `nginx -t`.

### 8-3. certbot 설치 (공식 pip 방식)
> AL2023 dnf 저장소엔 certbot이 없을 수 있어, certbot 공식 권장인 **격리된 venv + pip** 방식으로 설치한다.
```bash
sudo dnf install -y augeas-libs
sudo python3 -m venv /opt/certbot/
sudo /opt/certbot/bin/pip install --upgrade pip
sudo /opt/certbot/bin/pip install certbot certbot-nginx
sudo ln -sf /opt/certbot/bin/certbot /usr/bin/certbot
certbot --version              # 설치 확인
```

### 8-4. TLS 인증서 발급 (certbot이 443 설정 자동 주입)
```bash
sudo certbot --nginx -d trading.mydomain.com
# 이메일 입력 → 약관 동의 → "redirect(HTTP를 HTTPS로)" 선택
```
- certbot이 8-2의 80 블록에 **인증서 경로·443 listen·HTTP→HTTPS 리다이렉트를 자동으로 추가**한다.
- 끝나면 `https://trading.mydomain.com` 가 동작. 자동 갱신 테스트:
```bash
sudo certbot renew --dry-run
```
> 🔁 **자동 갱신엔 80이 다시 필요**하다. 80을 계속 닫아두면 90일 후 갱신이 실패하니, **80은 열어두되**(어차피 HTTPS로 리다이렉트만 함) 두거나, 갱신 직전에만 여는 방식을 택한다. 80을 열어두는 게 부담되면 DNS-01 갱신으로 전환 고려.

### 8-5. 접근 제한 강화 (IP 허용 + Basic Auth)
인증서가 들어가 443 블록이 생겼으니, 이제 **내 IP만 + 비밀번호**로 잠근다.
```bash
sudo dnf install -y httpd-tools                 # htpasswd 제공 (AL2023은 apache2-utils 아님)
sudo htpasswd -c /etc/nginx/.htpasswd admin     # admin 비밀번호 설정
sudo nano /etc/nginx/conf.d/stocktrading.conf
```
certbot이 만든 **`server { listen 443 ssl; ... }`** 블록 안, `location /` 위에 아래 4줄을 추가:
```nginx
    # 1차: 내 IP만 허용
    allow 193.186.4.167;
    deny all;
    # 2차: Basic Auth
    auth_basic           "Restricted";
    auth_basic_user_file /etc/nginx/.htpasswd;
```
적용:
```bash
sudo nginx -t && sudo systemctl reload nginx
```

### 8-6. 접근 확인 (TLS 경로일 때)
로컬 PC 브라우저(내 IP에서) `https://trading.mydomain.com/docs` → Basic Auth 창 → admin 로그인 → FastAPI 문서.
- 내 IP가 아니면 403, 비밀번호 틀리면 401 → 정상.

> 도메인 없이 운영하는 경우(우리 케이스)는 8-1~8-6을 쓰지 않고 **8-0 SSH 터널**로 접속한다.

## 9. 모니터링 · 운영 체크리스트

### 9-1. Lightsail 모니터링 & 비용 알림
- Lightsail → 인스턴스 → **Metrics**에서 CPU/네트워크 그래프 확인. CPU 알람(임계치) 설정 가능.
- AWS **Billing → Budgets**에서 월 예산 알림(예: $10 초과 시 메일) 설정 → 예상치 못한 과금 조기 감지.
- Lightsail은 **유휴 회수·재고 부족이 없으므로** 그쪽 걱정은 불필요(Oracle과 차이점).

### 9-2. 헬스 모니터링 / 죽음 감지 (트레이딩용 권장) 🟠
- 코드에 이미 **Slack 알림**(`SLACK_WEBHOOK_URL`)이 있으니, 시작/주문/에러 알림을 Slack으로 받도록 설정.
- 무인증 `/health` 엔드포인트를 별도로 두고(주문 라우터와 분리), 외부 모니터링(UptimeRobot 등)으로 핑하려면 nginx에서 `/health`만 `allow all` 예외 처리.
- `Restart=always`로 프로세스 크래시는 자동 복구되지만, **인스턴스 자체 중지/네트워크 장애**는 외부 감지가 필요.

### 9-4. 자주 겪는 트러블슈팅
| 증상 | 원인/조치 |
|------|-----------|
| SSH 접속 거부 | Lightsail 방화벽 22번 + 내 IP, 키 권한(`chmod 600`)·경로(`-i`), 사용자명 **`ec2-user`** 확인. 막히면 콘솔의 브라우저 SSH 사용 |
| 부팅 후 서비스 안 뜸 | `journalctl -u stocktrading -e`로 에러 확인, `.env`·venv 경로(`/home/ec2-user/...`) 점검 |
| 메모리 부족 크래시(OOM) | swap 적용/확대(5-3, `swapon --show`로 확인), 1GB 이상 플랜으로 업그레이드 |
| Supabase 쓰기 실패(403/RLS) | service_role 키 사용 여부 확인 (anon 키면 RLS에 막힘) |
| nginx 502 Bad Gateway | uvicorn(127.0.0.1:8000) 동작 확인 `curl http://127.0.0.1:8000/`, systemd status. **SELinux가 enforcing이면** `sudo setsebool -P httpd_can_network_connect 1` |
| certbot 발급 실패 | 도메인 A레코드가 `<서버IP>` 맞는지(`dig +short`), Lightsail 방화벽 80포트 임시 개방됐는지, nginx 80 블록 동작 중인지 확인 |
| `nginx -t` conflict / 시작 실패 | nginx.conf 기본 `listen 80 default_server` 블록과 충돌 → 주석 처리(8-2). 인증서 경로 오류면 8-4(certbot) 먼저 수행 |
| pip 설치 중 빌드 에러 | `sudo dnf install -y gcc python3.11-devel` 후 재시도 |
| IP 바뀌어 접속 불가 | 새 IP로 Lightsail 방화벽 갱신(5-2). firewalld도 쓴다면 rich rule 갱신 |

## 10. 비용 요약
- **Lightsail $5/월 (512MB, Dual-stack)** — 고정 요금, 유휴 회수·재고 변수 없음. 첫 3개월 무료 크레딧.
- 512MB는 **swap 2GB 전제**로 운영. 파이프라인 피크 때 swap 의존이 크면 1GB/2GB로 업그레이드(스냅샷 재생성).
- 고정 IP: 실행 인스턴스에 붙어 있으면 무료. 트래픽은 번들(1TB) 내라 추가 과금 사실상 없음.
- AWS Budgets로 월 예산 알림을 걸어두면 예상외 과금을 조기에 잡을 수 있다.
